# 第7章 原生 SQL 与 PostgreSQL 深度使用

第6章我们把 API 的"面子"做好了：路由、中间件、统一响应。但这一章开始前，数据还都存在内存里——服务一重启，报告就全丢了。真正的生产系统必须把数据持久化到数据库。

PostgreSQL 是这个世界上最好的开源关系数据库之一。它不仅稳定、功能丰富，还有 pgvector 扩展可以做向量检索（第12章详细讲）。我们这一章不讲 ORM，而是直接用 `database/sql` + `pgx` 写原生 SQL。**原因很实际**：ORM 在复杂查询和性能调优时会成为阻碍，原生 SQL 让你完全掌控 SQL 怎么执行。

## 7.1 database/sql + pgx：生产级连接配置

### 为什么选 pgx + database/sql

Go 访问 PostgreSQL 有两个主流选择：

1. **`lib/pq`**：老牌驱动，纯 Go，但官方已经宣布进入维护模式，新功能不再增加
2. **`jackc/pgx`**：目前最活跃、功能最完整的驱动，支持更多 PostgreSQL 特性（如 `COPY`、`LISTEN/NOTIFY`、批量操作）

我们选 `pgx`，但**不直接用 `pgx.Conn`**，而是通过 `pgx/stdlib` 拿到标准的 `*sql.DB`：

```go
import _ "github.com/jackc/pgx/v5/stdlib"

db, err := sql.Open("pgx", dsn)
```

这样做的好处：

- 代码依赖 `database/sql` 标准接口，未来换驱动成本极低
- 可以直接使用标准库的连接池、事务、Prepared Statement
- `pgx` 作为底层驱动，性能和兼容性更好

### 连接代码

我们在 `internal/repository/postgres/db.go` 中封装数据库连接：

```go
// 文件: src/backend/internal/repository/postgres/db.go

package postgres

import (
    "context"
    "database/sql"
    "fmt"
    "time"

    _ "github.com/jackc/pgx/v5/stdlib"
    "github.com/ileego/go_react_ai/internal/config"
)

// DB 封装数据库连接，提供连接池和迁移支持
type DB struct {
    *sql.DB
}

// New 使用配置创建 PostgreSQL 连接
func New(cfg config.DatabaseConfig) (*DB, error) {
    db, err := sql.Open("pgx", cfg.DSN())
    if err != nil {
        return nil, fmt.Errorf("open database: %w", err)
    }

    // 连接池配置：从配置文件读取，生产环境根据负载调整
    db.SetMaxOpenConns(cfg.MaxOpenConns)
    db.SetMaxIdleConns(cfg.MaxIdleConns)
    db.SetConnMaxLifetime(time.Duration(cfg.ConnMaxLifetime) * time.Minute)
    db.SetConnMaxIdleTime(time.Duration(cfg.ConnMaxIdleTime) * time.Minute)

    // 验证连接
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    if err := db.PingContext(ctx); err != nil {
        return nil, fmt.Errorf("ping database: %w", err)
    }

    return &DB{DB: db}, nil
}
```

注意几个细节：

- **`sql.Open` 不会真正连接数据库**，只是创建连接池对象，所以必须调用 `PingContext` 验证
- DSN 格式使用 PostgreSQL 标准：`postgres://user:password@host:port/db?sslmode=disable`
- `sslmode=disable` 只在开发环境使用，生产环境必须启用 SSL

DSN 来自 `config.DatabaseConfig.DSN()`，我们在第2章已经写好：

```go
func (d DatabaseConfig) DSN() string {
    return "postgres://" + d.User + ":" + d.Password +
        "@" + d.Host + ":" + d.Port + "/" + d.Name +
        "?sslmode=disable"
}
```

现在 `DatabaseConfig` 再增加连接池相关字段（`config.go` 中同步更新）：

```go
// 文件: src/backend/internal/config/config.go

type DatabaseConfig struct {

```

在 `Load()` 中设置默认值并读取：


### 启动时连接数据库

把连接代码串起来，`cmd/server/main.go` 在启动时创建数据库连接并执行迁移：

```go
// 文件: src/backend/cmd/server/main.go

func main() {
    cfg := config.Load()

    if cfg.Server.Mode == "release" {
        gin.SetMode(gin.ReleaseMode)
    }

    slog.SetDefault(slog.New(slog.NewJSONHandler(os.Stdout, nil)))

    // 连接 PostgreSQL 并自动执行迁移
    db, err := postgres.New(cfg.Database)
    if err != nil {
        log.Fatalf("Failed to connect database: %v", err)
    }
    defer db.Close()

    if err := postgres.MigrateUp(cfg.Database); err != nil {
        log.Fatalf("Failed to migrate database: %v", err)
    }

    // 依赖注入
    reportRepo := postgres.NewReportRepository(db.DB)
    reportSvc := service.NewReportService(reportRepo)
    agentSvc := service.NewAgentService(reportRepo, nil)

    handlers := handler.NewHandlers(reportSvc, agentSvc)
    // ... 中间件和路由注册
}
```

注意 `defer db.Close()`：服务退出时优雅关闭连接池，避免连接泄漏。

## 7.2 连接池调优：maxOpenConns、maxIdleConns、connMaxLifetime

### 四个核心参数

`database/sql` 的连接池有四个关键参数，理解它们是生产调优的基础：

| 参数 | 作用 | 默认值 | 建议 |
|------|------|--------|------|
| `SetMaxOpenConns` | 最大打开连接数 | 无限 | 根据 DB 最大连接数和并发量设置 |
| `SetMaxIdleConns` | 最大空闲连接数 | 2 | 通常设为 MaxOpenConns 的一半 |
| `SetConnMaxLifetime` | 连接最大复用时间 | 无限 | 30~60 分钟，配合数据库端超时 |
| `SetConnMaxIdleTime` | 连接最大空闲时间 | 无限 | 10~30 分钟，避免占着连接不用 |

### 一个常见误区

很多人把 `MaxOpenConns` 设得很大，认为"连接越多越好"。实际上：

- 每个连接都会消耗 PostgreSQL 的内存和 CPU
- 连接过多会导致数据库上下文切换开销剧增
- PostgreSQL 默认 `max_connections = 100`，你的应用只是其中一个客户端

**经验公式**：

```
单个应用实例的 MaxOpenConns ≈ (PostgreSQL max_connections - 预留) / 应用实例数
```

比如 PostgreSQL 设 `max_connections = 200`，有 4 个后端实例，每个实例大约 40~45 个连接。

### 我们的配置

连接池参数现在全部走配置文件，默认值如下：

```bash
DB_MAX_OPEN_CONNS=25
DB_MAX_IDLE_CONNS=10
DB_CONN_MAX_LIFETIME_MINUTES=30
DB_CONN_MAX_IDLE_TIME_MINUTES=10
```

开发环境这些值够用，生产部署时根据监控数据调整，不需要改代码、不需要重新编译，直接改环境变量或 `.env` 即可。

调参时建议关注三个指标：

1. **数据库活跃连接数**：`pg_stat_activity` 中你的应用连接数是否接近 `DB_MAX_OPEN_CONNS`
2. **等待连接的请求**：应用日志中是否出现 `sql: database is closed` 或长时间阻塞
3. **连接复用率**：空闲连接是否被频繁创建和销毁

## 7.3 SQL 组织最佳实践：常量包、查询构建器、命名参数

### 不要把 SQL 散落在业务代码里

初学者常犯的错误是把 SQL 直接写在 Handler 或 Service 里：

```go
// 反模式
func (h *Handler) Get(c *gin.Context) {
    row := db.QueryRow("SELECT id, title, topic, status, ... FROM reports WHERE id = $1", id)
    // ...
}
```

问题在于：
- SQL 改动时要到处找
- 重复字段容易写错
- 不利于复用和测试

我们把 SQL 集中在 Repository 文件底部作为常量：

```go
// 文件: src/backend/internal/repository/postgres/report_repository.go

const (
    queryReportByID = `
        SELECT id, title, topic, status, content, sources, created_by, created_at, updated_at, completed_at
        FROM reports
        WHERE id = $1
    `

    queryReportsByUser = `
        SELECT id, title, topic, status, content, sources, created_by, created_at, updated_at, completed_at
        FROM reports
        WHERE created_by = $1
        ORDER BY created_at DESC
        LIMIT $2 OFFSET $3
    `

    queryInsertReport = `
        INSERT INTO reports (title, topic, status, content, sources, created_by, created_at, updated_at, completed_at)
        VALUES ($1, $2, $3, $4, $5, $6, $7, $8, $9)
        RETURNING id
    `

    queryUpdateReportStatus = `
        UPDATE reports
        SET status = $1, updated_at = $2
        WHERE id = $3
    `
)
```

### PostgreSQL 数组的处理

我们的 `reports` 表有一个 `sources TEXT[]` 字段。Go 的 `database/sql` 不直接支持 PostgreSQL 数组类型，需要手动转换。

PostgreSQL 数组的文本格式是 `{a,b,c}`：

```go
// parsePGArray 解析 PostgreSQL 数组字符串
func parsePGArray(s string) []string {
    if s == "" || s == "{}" {
        return nil
    }
    s = strings.Trim(s, "{}")
    if s == "" {
        return nil
    }
    parts := strings.Split(s, ",")
    for i := range parts {
        parts[i] = strings.Trim(parts[i], "\"")
    }
    return parts
}

// formatPGArray 将 Go 字符串切片格式化为 PostgreSQL 数组字符串
func formatPGArray(items []string) string {
    if len(items) == 0 {
        return "{}"
    }
    var quoted []string
    for _, item := range items {
        quoted = append(quoted, fmt.Sprintf("\"%s\"", strings.ReplaceAll(item, "\"", "\\\"")))
    }
    return "{" + strings.Join(quoted, ",") + "}"
}
```

注意：这里用的是简化版解析。生产环境如果数组元素包含逗号或大括号，需要用更完善的解析器，比如 `jackc/pgtype`。

### 命名参数 vs 位置参数

PostgreSQL 使用 `$1, $2, $3` 作为位置参数。位置参数高效但不直观，维护时容易搞错顺序。

更友好的方式是使用命名参数库，比如 `sqlx` 的 `NamedQuery`：

```go
import "github.com/jmoiron/sqlx"

query := `SELECT * FROM reports WHERE created_by = :user_id AND status = :status`
rows, err := db.NamedQueryContext(ctx, query, map[string]any{
    "user_id": userID,
    "status":  status,
})
```

但 `sqlx` 会引入额外依赖。我们的项目目前不大，先用位置参数 + SQL 常量管理，等查询复杂了再引入 `sqlx`。

### Repository 层错误处理

引入 PostgreSQL 后，Repository 层的错误类型变得重要了。内存实现直接返回 `fmt.Errorf`，但 Service 层需要区分"资源不存在"和"数据库内部错误"。

我们在 `internal/repository` 包中定义统一的哨兵错误：

```go
// 文件: src/backend/internal/repository/repository.go

var ErrNotFound = errors.New("record not found")
```

Repository 实现中，遇到 `sql.ErrNoRows` 或查询无结果时返回 `repository.ErrNotFound`：

```go
// PostgreSQL 实现
if err == sql.ErrNoRows {
    return nil, repository.ErrNotFound
}

// 内存实现
if !ok {
    return nil, repository.ErrNotFound
}
```

Service 层根据错误类型决定返回什么业务错误：

```go
report, err := s.repo.GetByID(ctx, id)
if err != nil {
    if errors.Is(err, repository.ErrNotFound) {
        return nil, apperrors.NewNotFound("report", id)
    }
    return nil, apperrors.NewInternal("查询报告失败", err)
}
```

这样内部数据库错误不会被误报成 404，真正的 404 也不会被包装成 500。

## 7.4 事务与保存点：从简单事务到 Saga 模式

### 简单事务

数据库事务保证一组操作"要么全成功，要么全失败"。Go 中事务的标准写法：

```go
func (r *ReportRepository) CreateWithTasks(ctx context.Context, report *domain.Report) error {
    tx, err := r.db.BeginTx(ctx, nil)
    if err != nil {
        return err
    }
    defer tx.Rollback() // 安全网：提交失败时自动回滚

    // 插入报告
    if err := r.createInTx(ctx, tx, report); err != nil {
        return err
    }

    // 插入关联任务
    if err := r.createTasksInTx(ctx, tx, report.ID); err != nil {
        return err
    }

    return tx.Commit()
}
```

**关键模式**：
- `defer tx.Rollback()`：即使函数中途返回，也能保证未提交的事务被回滚
- 事务提交成功前，所有操作都在临时视图中
- 如果 `Commit()` 失败，必须返回错误，不能忽略

### 保存点（Savepoint）

保存点是事务内的"检查点"，可以回滚到某个点而不影响事务中的其他操作。适合长流程中的局部失败恢复：

```go
tx, _ := r.db.BeginTx(ctx, nil)
defer tx.Rollback()

// 创建保存点
if _, err := tx.ExecContext(ctx, "SAVEPOINT create_report"); err != nil {
    return err
}

// 尝试插入报告
if err := insertReport(tx); err != nil {
    // 回滚到保存点
    tx.ExecContext(ctx, "ROLLBACK TO SAVEPOINT create_report")
    // 但事务继续，可以记录日志或尝试替代方案
}

return tx.Commit()
```

### Saga 模式

Saga 是分布式长事务的解决方案。它不是数据库事务，而是把一个大事务拆成多个本地事务，每个本地事务有对应的"补偿操作"：

```
T1: 创建报告  → 成功
T2: 调用 AI 生成大纲  → 成功
T3: 保存大纲到数据库  → 失败
     ↓
C2: 取消 AI 任务
C1: 删除报告
```

Saga 不保证 ACID，但保证最终一致性。第25章实现多智能体调度时会用到这个模式。

## 7.5 数据库迁移：golang-migrate 团队协作规范

### 为什么需要迁移

数据库 schema 会随代码一起演进：加表、加字段、加索引。如果每个开发者都手动执行 SQL，很快就会失控：

- 测试环境的 schema 和生产不一致
- 新同事不知道数据库现在是什么状态
- 回滚时没有记录

**迁移（Migration）**就是把 schema 变更写成版本化脚本，让任何人都能从空数据库升到当前版本，也能安全回滚。

### golang-migrate

我们使用 `golang-migrate/migrate`，它是 Go 生态最流行的迁移工具。迁移文件命名格式：

```
{version}_{description}.{direction}.sql
```

例如：

```
000001_init.up.sql
000001_init.down.sql
000002_add_user_index.up.sql
000002_add_user_index.down.sql
```

- `.up.sql`：升级脚本
- `.down.sql`：回滚脚本
- 版本号必须递增，且保持固定长度（方便排序）

### 初始迁移

```sql
-- 文件: src/backend/internal/repository/migrations/000001_init.up.sql

CREATE TABLE IF NOT EXISTS users (
    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    nickname VARCHAR(100),
    role VARCHAR(50) NOT NULL DEFAULT 'user',
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 临时默认用户，第9章用户系统完成后移除
INSERT INTO users (id, email, password_hash, nickname, role)
VALUES (1, 'system@goai.local', 'notused', 'System', 'system')
ON CONFLICT (id) DO NOTHING;

CREATE TABLE IF NOT EXISTS reports (
    id BIGSERIAL PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    topic VARCHAR(500) NOT NULL,
    status VARCHAR(50) NOT NULL DEFAULT 'pending',
    content TEXT,
    sources TEXT[],
    created_by BIGINT NOT NULL REFERENCES users(id),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at TIMESTAMPTZ
);

CREATE TABLE IF NOT EXISTS agent_tasks (
    id BIGSERIAL PRIMARY KEY,
    report_id BIGINT NOT NULL REFERENCES reports(id) ON DELETE CASCADE,
    agent_role VARCHAR(50) NOT NULL,
    status VARCHAR(50) NOT NULL DEFAULT 'pending',
    input TEXT,
    output TEXT,
    cost_ms BIGINT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_reports_created_by ON reports(created_by);
CREATE INDEX IF NOT EXISTS idx_reports_status ON reports(status);
CREATE INDEX IF NOT EXISTS idx_agent_tasks_report_id ON agent_tasks(report_id);
```

对应的回滚脚本：

```sql
-- 文件: src/backend/internal/repository/migrations/000001_init.down.sql

DROP INDEX IF EXISTS idx_agent_tasks_report_id;
DROP INDEX IF EXISTS idx_reports_status;
DROP INDEX IF EXISTS idx_reports_created_by;

DROP TABLE IF EXISTS agent_tasks;
DROP TABLE IF EXISTS reports;
DROP TABLE IF EXISTS users;
```

### 在 Go 代码中执行迁移

我们在 `main.go` 启动时自动执行迁移，避免手动操作：

```go
// 文件: src/backend/internal/repository/postgres/migrate.go

func MigrateUp(cfg config.DatabaseConfig) error {
    db, err := sql.Open("pgx", cfg.DSN())
    if err != nil {
        return err
    }
    defer db.Close()

    driver, err := pgx.WithInstance(db, &pgx.Config{})
    if err != nil {
        return fmt.Errorf("create migrate driver: %w", err)
    }

    m, err := migrate.NewWithDatabaseInstance(
        "file://internal/repository/migrations",
        "pgx", driver)
    if err != nil {
        return fmt.Errorf("create migrate instance: %w", err)
    }

    if err := m.Up(); err != nil && err != migrate.ErrNoChange {
        return fmt.Errorf("migrate up: %w", err)
    }
    return nil
}
```

注意 `migrate.ErrNoChange`：如果数据库已经是最高版本，调用 `Up()` 会返回这个错误，需要忽略它。

### Makefile 命令

```makefile
# 文件: src/backend/Makefile

migrate:
    migrate -path internal/repository/migrations \
        -database "postgres://goai:goai_dev@localhost:5432/goai?sslmode=disable" up

migrate-new:
    @if [ -z "$(NAME)" ]; then \
        echo "用法: make migrate-new NAME=xxx"; \
        exit 1; \
    fi
    migrate create -ext sql -dir internal/repository/migrations $(NAME)
```

### 团队协作规范

迁移脚本是团队协作的关键，必须遵守以下规则：

1. **每个 schema 变更一个迁移文件**：不要把多个不相关的改动放进一个文件
2. **迁移文件一旦提交，不要修改**：如果发现错误，新建一个迁移文件修复
3. **`.down.sql` 必须可执行**：虽然生产环境很少回滚，但测试环境经常需要
4. **迁移前在测试数据库验证**：`make migrate` 前先 `make up` 启动本地数据库
5. **不要在迁移里放业务数据**：只有 schema 变更和必要的静态数据（如枚举值）
6. **大表加索引用 `CONCURRENTLY`**：避免锁表，但 `CONCURRENTLY` 不能在事务块中执行，需要单独处理

## 7.6 索引设计：B-tree、GIN、GiST 适用场景

### B-tree：默认且最常用的索引

B-tree 适合等值查询和范围查询，是 PostgreSQL 默认的索引类型：

```sql
CREATE INDEX idx_reports_created_by ON reports(created_by);
CREATE INDEX idx_reports_status ON reports(status);
```

适用场景：
- `WHERE id = 1`
- `WHERE created_at > '2026-06-01'`
- `ORDER BY created_at DESC`
- 外键字段

### GIN：数组和全文搜索

GIN（Generalized Inverted Index）适合包含多个值的字段，比如数组、JSONB、全文搜索：

```sql
-- 搜索 sources 数组包含某个 URL 的报告
CREATE INDEX idx_reports_sources_gin ON reports USING GIN(sources);

SELECT * FROM reports WHERE sources @> ARRAY['https://example.com'];
```

适用场景：
- 数组字段查询
- JSONB 字段的键值查询
- PostgreSQL 全文搜索（`tsvector`）

### GiST：几何和相似度

GiST（Generalized Search Tree）适合空间数据和相似度搜索。我们在第12章用它做向量索引：

```sql
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops);
```

HNSW 是 pgvector 提供的一种近似最近邻索引，基于 GiST/GIN 的扩展机制。

### 索引不是越多越好

每加一个索引，写入性能就会下降一点：

- `INSERT` 需要维护所有索引
- `UPDATE` 涉及索引字段时也要更新索引
- 索引占用磁盘空间

**索引设计原则**：

1. 先跑查询，再建索引
2. 优先给 WHERE、JOIN、ORDER BY 的字段建索引
3. 高频写入的表少建索引
4. 定期用 `pg_stat_user_indexes` 检查无效索引

## 7.7 查询性能分析：EXPLAIN ANALYZE 解读与优化

### EXPLAIN ANALYZE 基础

`EXPLAIN ANALYZE` 是 PostgreSQL 最强大的性能分析工具，它会显示查询的执行计划和实际耗时：

```sql
EXPLAIN ANALYZE
SELECT * FROM reports
WHERE created_by = 1
ORDER BY created_at DESC
LIMIT 20;
```

输出类似：

```
Limit  (cost=0.29..8.30 rows=20 width=200) (actual time=0.023..0.045 rows=20 loops=1)
  ->  Index Scan Backward using idx_reports_created_at on reports  (cost=0.29..123.45 rows=300 width=200) (actual time=0.021..0.040 rows=20 loops=1)
        Filter: (created_by = 1)
Planning Time: 0.150 ms
Execution Time: 0.060 ms
```

### 关键指标解读

| 指标 | 含义 | 注意点 |
|------|------|--------|
| `cost` | 优化器估算的成本 | 只看相对大小，不看绝对值 |
| `actual time` | 实际执行时间 | 第一个数字是首行时间，第二个是总时间 |
| `rows` | 估算 / 实际返回行数 | 估算偏差大说明统计信息过期 |
| `loops` | 执行次数 | 嵌套循环中这个数字会很大 |

### 常见坏味道

1. **Seq Scan（全表扫描）**：小表没问题，大表需要加索引
2. **Nested Loop 嵌套循环**：连接条件缺少索引时会变慢
3. **Sort 内存不足**：`work_mem` 太小会导致磁盘排序
4. **Rows 估算偏差大**：需要 `ANALYZE` 更新统计信息

### 优化实例

假设查询报告列表很慢：

```sql
SELECT * FROM reports
WHERE created_by = 1
ORDER BY created_at DESC
LIMIT 20;
```

`EXPLAIN ANALYZE` 显示 Seq Scan。优化：

```sql
CREATE INDEX idx_reports_user_created
ON reports(created_by, created_at DESC);
```

这个复合索引同时覆盖 WHERE 和 ORDER BY，查询可以直接走索引，不需要回表排序。

### 定期维护

PostgreSQL 需要定期维护才能保持性能：

```sql
-- 更新统计信息
ANALYZE reports;

-- 清理死元组，回收空间
VACUUM ANALYZE reports;

-- 重建索引（较少需要）
REINDEX INDEX idx_reports_user_created;
```
