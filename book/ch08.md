# 第8章 错误处理、日志与可观测性

第7章我们把数据持久化到了 PostgreSQL，但"能跑"和"能运维"是两回事。线上系统出问题时的第一反应往往是：日志在哪里？错误码是什么意思？服务到底还健康吗？

这一章我们补齐后端的生产级可观测性能力：统一的业务错误码、结构化的 `slog` 日志、可配置的日志级别、请求 ID 全链路透传，以及区分存活和就绪的健康检查。 Prometheus 指标和分布式链路追踪会在第 31 章深入，这里先把最基础、最高 ROI 的能力落地。

## 8.1 Go 错误处理哲学：显式优于隐式

Go 的错误处理常被吐槽"满屏 `if err != nil`"。但这恰恰是一种工程上的诚实：**每个可能失败的点都显式处理**，调用方不会突然收到一个从深处抛出的异常。问题的关键不是 `if err != nil` 太多，而是错误传递过程中信息不能丢。

我们的分层是 Handler → Service → Repository。上层调用下层时，有两种常见错误：

1. **业务错误**：参数不合法、资源不存在、权限不足——前端需要明确感知
2. **内部错误**：数据库连接失败、SQL 语法错误——不应该直接暴露给前端，但要记录完整上下文

早在第 3 章，我们就已经在 `pkg/errors` 中定义了统一错误类型；第 6 章 `pkg/response.FromError` 实现了按 `Kind` 映射 HTTP 状态码；第 7 章末尾则在 Repository 层引入了 `ErrNotFound` 哨兵错误，并在 Service 层通过 `errors.Is` 把它转换成业务错误。这一章先把错误体系补全。

### 补全错误类型和构造函数

```go
// 文件: src/backend/pkg/errors/errors.go

package errors

import "fmt"

type Kind string

const (
    KindValidation   Kind = "validation"
    KindNotFound     Kind = "not_found"
    KindDuplicate    Kind = "duplicate"
    KindInternal     Kind = "internal"
    KindUnauthorized Kind = "unauthorized"
    KindForbidden    Kind = "forbidden"
)

type Error struct {
    Kind    Kind
    Code    string // 前端稳定错误码，如 REPORT_NOT_FOUND
    Field   string // 校验错误时对应的字段
    Message string
    Cause   error  // 原始错误，用于日志追溯
}

func (e *Error) Error() string { /* ... */ }

func (e *Error) Unwrap() error { return e.Cause }

// WithCode 为错误设置前端稳定错误码，支持链式调用
func (e *Error) WithCode(code string) *Error {
    e.Code = code
    return e
}

func NewValidation(field, message string) *Error { /* ... */ }
func NewNotFound(resource string, id any) *Error { /* ... */ }
func NewInternal(message string, cause error) *Error { /* ... */ }

// 新增构造函数
func NewDuplicate(resource string, id any) *Error {
    return &Error{Kind: KindDuplicate, Message: fmt.Sprintf("%s (id=%v) 已存在", resource, id)}
}

func NewUnauthorized(message string) *Error {
    return &Error{Kind: KindUnauthorized, Message: message}
}

func NewForbidden(message string) *Error {
    return &Error{Kind: KindForbidden, Message: message}
}
```

这里新增了一个 `Code` 字段，它不是 HTTP 状态码，而是给前端用的业务错误码。比如"报告不能取消"可以定义为 `REPORT_CANNOT_CANCEL`，前端拿到这个码就能展示对应的本地化文案，而不必解析 HTTP 消息文本。

## 8.2 错误链与包装：errors.Is、errors.As 的正确用法

Go 1.13 引入的 `%w` 和 `errors.Is`/`errors.As` 让错误链变得清晰。我们的 `pkg/errors.Error` 实现了 `Unwrap()`，所以可以这样用：

```go
// 文件: src/backend/internal/service/report_service.go

import (
    "context"
    "errors"

    "github.com/ileego/go_react_ai/internal/repository"
    apperrors "github.com/ileego/go_react_ai/pkg/errors"
)

func (s *reportService) GetByID(ctx context.Context, id int64) (*domain.Report, error) {
    report, err := s.repo.GetByID(ctx, id)
    if err != nil {
        if errors.Is(err, repository.ErrNotFound) {
            return nil, apperrors.NewNotFound("report", id)
        }
        // 内部错误保留原始错误作为 Cause，方便日志追溯
        return nil, apperrors.NewInternal("查询报告失败", err)
    }
    return report, nil
}
```

关键点：

- **Repository 层**用哨兵错误 `repository.ErrNotFound` 标识"记录不存在"这一事实
- **Service 层**用 `errors.Is` 判断后，包装成带业务语义的 `apperrors.Error`
- **内部错误**通过 `Cause` 字段保留原始错误，日志排查时能看到完整堆栈链；业务错误（如 NotFound）由 Service 层根据上下文重新构造，通常不需要 Cause

### 反模式：直接返回裸错误

```go
// 不要这样做
if err != nil {
    return err  // 上层不知道这是业务错误还是内部错误
}
```

正确做法：每一层都把错误包装成对当前层有意义的类型，同时保留原始错误。

## 8.3 业务错误码体系：给前端可处理的错误标识

HTTP 状态码只能表达错误大类。比如 400 可能代表"标题为空"、"标题太长"、"状态不允许取消"等多种情况。前端如果靠解析 `message` 做判断，一旦文案调整就失效。

我们在统一响应体中增加 `err_code`：

```go
// 文件: src/backend/pkg/response/response.go

package response

import (
    "net/http"

    "github.com/gin-gonic/gin"
    "github.com/ileego/go_react_ai/pkg/errors"
)

type Body[T any] struct {
    Code    int    `json:"code"`
    ErrCode string `json:"err_code,omitempty"`
    Message string `json:"message,omitempty"`
    Data    T      `json:"data,omitempty"`
}

var defaultErrCode = map[errors.Kind]string{
    errors.KindValidation:   "VALIDATION_ERROR",
    errors.KindNotFound:     "NOT_FOUND",
    errors.KindDuplicate:    "DUPLICATE",
    errors.KindUnauthorized: "UNAUTHORIZED",
    errors.KindForbidden:    "FORBIDDEN",
    errors.KindInternal:     "INTERNAL_ERROR",
}

// FromError 根据业务错误类型返回对应的 HTTP 响应
func FromError(c *gin.Context, err error) {
    if e, ok := err.(*errors.Error); ok {
        status := http.StatusInternalServerError
        switch e.Kind {
        case errors.KindValidation:
            status = http.StatusBadRequest
        case errors.KindNotFound:
            status = http.StatusNotFound
        case errors.KindDuplicate:
            status = http.StatusConflict
        case errors.KindUnauthorized:
            status = http.StatusUnauthorized
        case errors.KindForbidden:
            status = http.StatusForbidden
        }

        errCode := e.Code
        if errCode == "" {
            errCode = defaultErrCode[e.Kind]
        }
        c.JSON(status, Body[any]{Code: status, ErrCode: errCode, Message: e.Message})
        return
    }
    c.JSON(http.StatusInternalServerError, Body[any]{
        Code:    http.StatusInternalServerError,
        ErrCode: "INTERNAL_ERROR",
        Message: err.Error(),
    })
}
```

### 错误码映射表

| Kind | HTTP 状态码 | 默认 ErrCode | 使用场景 |
|------|------------|--------------|---------|
| validation | 400 | VALIDATION_ERROR | 参数校验失败 |
| not_found | 404 | NOT_FOUND | 资源不存在 |
| duplicate | 409 | DUPLICATE | 重复数据 |
| unauthorized | 401 | UNAUTHORIZED | 未登录/Token 无效 |
| forbidden | 403 | FORBIDDEN | 无权限 |
| internal | 500 | INTERNAL_ERROR | 服务器内部错误 |

### 自定义错误码

对于需要前端精确处理的场景，使用 `WithCode`：

```go
// 文件: src/backend/internal/service/report_service.go

if !report.CanCancel() {
    return apperrors.NewValidation("status", fmt.Sprintf("当前状态 %s 不允许取消", report.Status)).
        WithCode("REPORT_CANNOT_CANCEL")
}
```

前端收到：

```json
{
  "code": 400,
  "err_code": "REPORT_CANNOT_CANCEL",
  "message": "当前状态不允许取消"
}
```

前端可以直接根据 `err_code` 弹出"无法取消"的专属提示，而不依赖后端文案。

## 8.4 结构化日志：slog 标准库与 JSON 输出

日志是排查线上问题的第一手资料。但 `log.Printf` 打印出来的是自由文本，比如：

```
2026/06/15 13:58:23 GET /api/reports 200 12.345ms
```

这条日志人眼能读，但日志平台很难按字段过滤：你想统计所有 500 错误的请求，得写正则；想按 `request_id` 检索，也得写正则。规模一上来，这种非结构化日志几乎无法分析。

所以我们使用从 Go 1.21 开始加入标准库的 `log/slog`。`slog` 的核心思想是**键值对结构化**：一条日志不再是字符串，而是一组带类型的字段。上面的请求日志用 `slog` 输出后变成：

```json
{
  "time": "2026-06-15T13:58:23.123456789+08:00",
  "level": "INFO",
  "msg": "http request",
  "method": "GET",
  "path": "/api/reports",
  "status": 200,
  "latency": "12.345ms",
  "ip": "127.0.0.1",
  "request_id": "a1b2c3d4",
  "errors": 0
}
```

每条信息都有独立字段，日志平台可以直接按 `status=500` 或 `request_id=xxx` 查询，排查效率提升一个数量级。

### 初始化 slog

第 6 章时我们已经在 `main.go` 里写死了 JSON 日志。现在把它抽成可配置的 `initLogger`，让级别和格式都能通过环境变量调整：

```go
// 文件: src/backend/cmd/server/main.go

package main

import (
    "log/slog"
    "os"
    // ...
)

func main() {
    cfg := config.Load()

    // 初始化结构化日志：级别和格式均可通过环境变量配置
    initLogger(cfg.Server.LogLevel, cfg.Server.LogFormat)

    // ...
}

// initLogger 根据配置初始化 slog，支持运行时调整日志级别
func initLogger(levelStr, format string) {
    var level slog.LevelVar
    if err := level.UnmarshalText([]byte(levelStr)); err != nil {
        level.Set(slog.LevelInfo)
    }

    opts := &slog.HandlerOptions{Level: &level}
    var handler slog.Handler
    switch format {
    case "text":
        handler = slog.NewTextHandler(os.Stdout, opts)
    default:
        handler = slog.NewJSONHandler(os.Stdout, opts)
    }
    slog.SetDefault(slog.New(handler))
}
```

注意我们使用 `slog.LevelVar` 而不是固定的 `slog.LevelInfo`。`LevelVar` 的值可以在运行时通过 `level.Set(slog.LevelDebug)` 修改，这对线上临时开启 DEBUG 非常有用。

### 什么时候用 JSON，什么时候用文本

| 场景 | 推荐格式 | 原因 |
|------|---------|------|
| 本地开发 | `text` | 人眼阅读更直观 |
| 测试/CI | `text` | 失败日志直接可读 |
| 生产环境 | `json` | 便于日志平台解析和检索 |

本地开发时可以这样启动：

```bash
LOG_LEVEL=debug LOG_FORMAT=text go run ./cmd/server
```

生产环境保持默认值即可：

```bash
LOG_LEVEL=info LOG_FORMAT=json go run ./cmd/server
```

## 8.5 日志分级策略：开发 DEBUG、生产 INFO+ 的实践

日志级别不是越多越好，生产环境一般遵守以下原则：

| 级别 | 使用场景 | 生产环境是否输出 |
|------|---------|----------------|
| DEBUG | 详细的调试信息，如 SQL 参数、请求体 | 否 |
| INFO | 关键路径信息，如请求完成、服务启动 | 是 |
| WARN | 非致命异常，如重试、降级 | 是 |
| ERROR | 需要立即处理的问题，如数据库连接失败 | 是 |

我们通过环境变量控制级别：

```bash
# 开发环境
LOG_LEVEL=debug

# 生产环境
LOG_LEVEL=info
```

在 `config.go` 中增加配置：

```go
// 文件: src/backend/internal/config/config.go

type ServerConfig struct {
    Port         string `mapstructure:"SERVER_PORT"`
    Mode         string `mapstructure:"SERVER_MODE"`
    AllowOrigins string `mapstructure:"ALLOW_ORIGINS"`
    LogLevel     string `mapstructure:"LOG_LEVEL"`  // debug / info / warn / error
    LogFormat    string `mapstructure:"LOG_FORMAT"` // json / text
}
```

并在 `Load()` 中设置默认值：

```go
v.SetDefault("LOG_LEVEL", "info")
v.SetDefault("LOG_FORMAT", "json")
```

`.env.example` 同步更新：

```bash
# 文件: src/backend/.env.example

SERVER_PORT=8080
SERVER_MODE=debug
ALLOW_ORIGINS=*
LOG_LEVEL=info
LOG_FORMAT=json
```

### 使用示例

代码中按级别记录：

```go
slog.Debug("开始查询报告列表", "user_id", userID, "page", page)
slog.Info("报告创建成功", "report_id", report.ID)
slog.Warn("数据库连接池接近上限", "open_conns", open, "max_open", max)
slog.Error("查询报告失败", "error", err)
```

生产环境只输出 INFO 及以上，避免 DEBUG 日志淹没关键信息。

## 8.6 分布式追踪：请求 ID 全链路透传

当系统有多个服务时，一次用户请求会经过网关、API、数据库、缓存、外部 API。如果每个服务各自打印日志，排查问题时很难把散落各处的日志串起来。**请求 ID（Request ID）**就是解决这个问题最简单有效的手段。

我们在第 6 章已经实现了 `RequestID` 中间件，这一章把它和 `slog` 打通。

### RequestID 中间件

```go
// 文件: src/backend/internal/middleware/request_id.go

const HeaderXRequestID = "X-Request-ID"

func RequestID() gin.HandlerFunc {
    return func(c *gin.Context) {
        rid := c.GetHeader(HeaderXRequestID)
        if rid == "" {
            rid = generateID()
        }
        c.Set(HeaderXRequestID, rid)
        c.Writer.Header().Set(HeaderXRequestID, rid)
        c.Next()
    }
}
```

如果客户端不传，服务端生成；如果客户端传了，就复用。响应头也会把 `X-Request-ID` 带回去，方便客户端排查。

### 把 Logger 注入上下文

请求 ID 只是标识，真正让日志串起来的是**把带 request_id 的 logger 放进 gin 上下文**。注入点就在 `Logger()` 中间件里：

```go
// 文件: src/backend/internal/middleware/logger.go

func Logger() gin.HandlerFunc {
    return func(c *gin.Context) {
        start := time.Now()
        path := c.Request.URL.Path
        raw := c.Request.URL.RawQuery

        // 把带 request_id 的 logger 注入上下文
        logger := slog.With(slog.String("request_id", GetRequestID(c)))
        c.Set("logger", logger)

        c.Next()

        latency := time.Since(start)
        status := c.Writer.Status()
        if raw != "" {
            path = path + "?" + raw
        }

        logger.Info("http request",
            slog.String("method", c.Request.Method),
            slog.String("path", path),
            slog.Int("status", status),
            slog.Duration("latency", latency),
            slog.String("ip", c.ClientIP()),
            slog.Int("errors", len(c.Errors)),
        )
    }
}
```

注意 `Logger()` 必须在 `RequestID()` 之后注册，否则拿不到 request_id。`main.go` 里的顺序是：

```go
r.Use(middleware.Recovery())  // 最外层，捕获所有 panic
r.Use(middleware.RequestID()) // 先生成 request_id
r.Use(middleware.Logger())    // 再注入带 request_id 的 logger
r.Use(middleware.CORS(cfg.Server.AllowOrigins))
```

这样即使后续 Handler 发生 panic，`Recovery()` 中间件也能通过 `GetLogger(c)` 拿到已经注入的 logger，panic 日志同样带 request_id。

### 获取上下文 Logger

新建 `internal/middleware/context.go`：

```go
// 文件: src/backend/internal/middleware/context.go

package middleware

import (
    "log/slog"

    "github.com/gin-gonic/gin"
)

const loggerContextKey = "logger"

// GetLogger 从 gin.Context 获取注入的 logger；若不存在则返回默认 logger
func GetLogger(c *gin.Context) *slog.Logger {
    if v, ok := c.Get(loggerContextKey); ok {
        if l, ok := v.(*slog.Logger); ok {
            return l
        }
    }
    return slog.Default()
}
```

Handler 或 Service 里可以这样用：

```go
logger := middleware.GetLogger(c)
logger.Info("开始派发报告任务", "report_id", reportID)
```

这样任何一条日志都会自动带上 `request_id`，排查问题时用 `request_id` 一搜即可串起整条链路。

### Panic 恢复也带请求 ID

```go
// 文件: src/backend/internal/middleware/recovery.go

func Recovery() gin.HandlerFunc {
    return func(c *gin.Context) {
        defer func() {
            if err := recover(); err != nil {
                logger := GetLogger(c)
                logger.Error("panic recovered",
                    slog.Any("error", err),
                    slog.String("path", c.Request.URL.Path),
                    slog.String("method", c.Request.Method),
                )
                response.InternalServerError(c, "服务器内部错误")
                c.Abort()
            }
        }()
        c.Next()
    }
}
```

## 8.7 健康检查、就绪探针与性能指标暴露

Kubernetes 区分两种探针：

- **Liveness（存活探针）**：进程是否还在运行。如果失败，K8s 会重启容器。
- **Readiness（就绪探针）**：服务是否准备好接收流量。如果失败，K8s 会把 Pod 从 Service 端点中移除。

我们用 `/api/health` 做存活检查，`/api/ready` 做就绪检查。

### 健康检查处理器

```go
// 文件: src/backend/internal/handler/health_handler.go

package handler

import (
    "context"
    "net/http"
    "time"

    "github.com/gin-gonic/gin"
    "github.com/ileego/go_react_ai/internal/middleware"
    "github.com/ileego/go_react_ai/pkg/response"
)

type HealthHandler struct {
    dbHealth func(context.Context) error
}

func NewHealthHandler(dbHealth func(context.Context) error) *HealthHandler {
    return &HealthHandler{dbHealth: dbHealth}
}

// Check 存活探针
func (h *HealthHandler) Check(c *gin.Context) {
    response.Data(c, gin.H{
        "status": "ok",
        "time":   time.Now().Unix(),
    })
}

// Ready 就绪探针
func (h *HealthHandler) Ready(c *gin.Context) {
    if h.dbHealth == nil {
        response.Data(c, gin.H{"status": "ready", "time": time.Now().Unix()})
        return
    }

    ctx, cancel := context.WithTimeout(c.Request.Context(), 3*time.Second)
    defer cancel()

    if err := h.dbHealth(ctx); err != nil {
        logger := middleware.GetLogger(c)
        logger.Error("readiness check failed", "error", err.Error())
        c.JSON(http.StatusServiceUnavailable, gin.H{
            "status": "not_ready",
            "reason": "database_unavailable",
        })
        return
    }

    response.Data(c, gin.H{"status": "ready", "time": time.Now().Unix()})
}
```

`dbHealth` 使用 `postgres.DB.HealthCheck`：

```go
// 文件: src/backend/internal/repository/postgres/db.go

func (d *DB) HealthCheck(ctx context.Context) error {
    return d.PingContext(ctx)
}
```

### 注册路由

```go
// 文件: src/backend/internal/handler/handler.go

func NewHandlers(
    reportSvc service.ReportService,
    agentSvc service.AgentService,
    dbHealth func(context.Context) error,
) *Handlers {
    return &Handlers{
        Report: NewReportHandler(reportSvc),
        Agent:  NewAgentHandler(agentSvc),
        Health: NewHealthHandler(dbHealth),
    }
}

func (h *Handlers) RegisterRoutes(r *gin.Engine) {
    api := r.Group("/api")
    {
        api.GET("/health", h.Health.Check)
        api.GET("/ready", h.Health.Ready)
        // ...
    }
}
```

在 `main.go` 中注入：

```go
handlers := handler.NewHandlers(reportSvc, agentSvc, db.HealthCheck)
```

### 探针设计原则

| 探针 | 路径 | 失败含义 | 失败动作 |
|------|------|---------|---------|
| Liveness | /api/health | 进程卡死或 panic | 重启容器 |
| Readiness | /api/ready | 依赖不可用（如 DB） | 停止流量转发 |

**不要把重量级检查放进 Liveness**。如果数据库抖动导致 Liveness 失败，K8s 会不断重启容器，反而加剧问题。耗时的、依赖外部服务的检查（如数据库 ping、Redis 连接、第三方 API 探测）只应该放在 Readiness 里。

这一章把后端的"可观测性三支柱"之一——日志和错误处理——夯实了：

- **错误体系**：`pkg/errors` 补全构造函数和前端稳定错误码 `Code`
- **响应体**：`pkg/response.Body` 增加 `err_code`，`FromError` 按 `Kind` 映射状态码
- **日志配置**：`LOG_LEVEL` / `LOG_FORMAT` 支持环境变量配置，使用 `slog.LevelVar`
- **上下文日志**：`RequestID` + `GetLogger(c)` 让每条日志都带 `request_id`
- **健康检查**：`/api/health` 存活探针 + `/api/ready` 就绪探针，DB 不正常时返回 503
- **废弃清理**：删除 `internal/domain/errors.go`，避免与 `pkg/errors` 重复

启动服务验证：

```bash
$ cd src/backend
$ go run ./cmd/server
```

另开终端测试：

```bash
$ curl -i http://localhost:8080/api/health
{"code":0,"data":{"status":"ok","time":...}}

$ curl -i http://localhost:8080/api/ready
{"code":0,"data":{"status":"ready","time":...}}

$ curl -i -H "X-Request-ID: debug-123" http://localhost:8080/api/reports/999999
{"code":404,"err_code":"NOT_FOUND","message":"report (id=999999) 不存在"}
```

查看服务端日志，会发现所有日志都包含 `"request_id":"debug-123"`。
