# 第5章 AI 辅助开发：让 AI 成为你的结对编程伙伴

写代码这件事，正在经历一场静悄悄的革命。

不是 AI 要取代程序员，而是会用 AI 的程序员正在取代不会用的。这不是危言耸听——当你看到同事用 Cursor 十分钟生成一个你手写需要两小时的模块，或者用 Claude Code 一句话完成跨五个文件的重构时，你会意识到：**开发效率的基准线已经被抬高了**。

这一章我们不讲理论，只讲实战。你会学到怎么把 AI 工具嵌入到开发流程的每一个环节，从需求分析到代码审查，从原型设计到生产调试。更重要的是，你会知道这些工具的边界在哪里——什么时候该信任 AI，什么时候必须人工把关。

## 5.1 AI 编程助手全景：谁适合干什么

市场上的 AI 编程助手已经超过二十款，但真正值得日常使用的只有五款。我们不评测所有功能，只看各自最擅长的场景。

### Cursor：全栈开发者的首选

[Cursor](https://cursor.com) 是基于 VS Code 的 AI 编辑器，内置了 GPT-4 / Claude 的能力。它的核心竞争力是**代码库级上下文理解**。

**为什么选它**：
- `@` 符号引用：可以 `@` 某个文件、某个函数、某个目录，让 AI 精确理解你要改什么
- `Ctrl+K` Inline Edit：选中代码，直接说"给这个函数加错误处理"，AI 原地修改
- `Ctrl+L` Chat：针对当前文件提问，AI 能看到你的代码
- Composer 模式：多文件同时编辑，AI 自动协调跨文件变更

**最佳场景**：
- 写新功能（特别是跨前后端的）
- 理解陌生代码库
- 批量重构

**价格**：Pro 版 $20/月，对企业开发来说性价比极高。

### GitHub Copilot：代码补全最强

Copilot 在你打字时自动建议下一行、下一个函数。它的补全不是基于模板，而是基于整个代码库的上下文。

**为什么选它**：
- 补全质量最高，特别是 boilerplate 代码
- 与 VS Code / JetBrains 深度集成，零学习成本
- Copilot Chat 可以解释代码、生成测试

**最佳场景**：
- 日常编码、补全
- 写重复性代码（结构体定义、接口实现、测试用例）
- 快速生成注释

**价格**：$10/月（个人），$19/月（企业）。

### Claude Code：终端里的架构师

Claude Code 是 Anthropic 推出的终端 AI 助手。与 Cursor 和 Copilot 不同，它不是在 IDE 里，而是在终端里运行。

```bash
$ claude
> 帮我给所有 API handler 加上请求日志中间件
> 找出这个项目里所有没有错误处理的地方
> 把这个函数重构为使用 context 传递超时
```

**为什么选它**：
- 批量修改能力最强（可以一次改十几个文件）
- 代码审查和架构分析能力突出
- 不需要打开 IDE，适合服务器环境

**最佳场景**：
- 批量重构
- 代码审查
- 项目健康检查（找出技术债务、重复代码、潜在 bug）

**价格**：包含在 Claude Pro 订阅中（$20/月）。

### WindSurf：Cascade 工作流

[WindSurf](https://codeium.com/windsurf) 是 Codeium 团队的新产品，主打 **Cascade** 工作流——AI  agent 可以自主执行多步骤任务，比如"给这个项目加上用户认证系统"，它会自己创建文件、修改代码、运行测试。

**为什么选它**：
- Agent 模式可以自动执行多步骤任务
- 对开源项目免费

**最佳场景**：
- 快速搭建 MVP
- 自动化重复性任务

### Codeium：免费替代方案

[Codeium](https://codeium.com) 提供免费的代码补全和聊天功能，对开源项目和个人开发者非常友好。

**最佳场景**：预算有限的个人开发者、开源项目贡献者。

### 选型决策表

| 场景 | 首选 | 备选 |
|------|------|------|
| 日常编码补全 | Copilot | Codeium |
| 写新功能/跨文件修改 | Cursor | WindSurf |
| 批量重构/代码审查 | Claude Code | Cursor |
| 快速原型/MVP | WindSurf | Cursor |
| 预算敏感 | Codeium | 开源模型本地部署 |

我们的建议是：**Copilot + Cursor + Claude Code 三件套**。Copilot 处理日常补全，Cursor 处理功能开发，Claude Code 处理批量重构和审查。每月 $50 的订阅费，节省的时间价值远超这个数。

## 5.2 工具选型与配置：不同场景下的最优组合

### IDE 配置：VS Code + Cursor 扩展

如果你已经习惯了 VS Code，迁移到 Cursor 几乎零成本。安装后按 `Ctrl+Shift+P`，输入 "Cursor: Sign In" 登录即可。

推荐配置（`.vscode/settings.json`）：

```json
{
  "cursor.aIenableAutoImport": true,
  "cursor.aIenableInlineEdit": true,
  "editor.quickSuggestions": {
    "other": true,
    "comments": false,
    "strings": false
  }
}
```

### Claude Code 配置

```bash
# 安装
$ npm install -g @anthropic-ai/claude-code

# 在项目目录启动
$ claude

# 配置忽略文件（类似 .gitignore）
$ cat > .claudeignore << 'EOF'
node_modules/
dist/
*.log
EOF
```

`.claudeignore` 很重要——Claude Code 会索引整个代码库，忽略不需要的文件可以显著减少 token 消耗和响应时间。

### 多工具协作工作流

我们的日常开发流程是这样的：

```
需求分析        → Cursor Chat / Claude Code
架构设计        → Cursor Composer / Claude Code
原型设计        → Cursor + Vite 实时预览
编码            → Copilot 补全 + Cursor Inline Edit
调试            → Claude Code 分析日志
测试            → Copilot 生成用例 + Cursor 补充边界
重构            → Claude Code 批量修改
代码审查        → Claude Code 审查 + 人工确认
文档            → Cursor 生成注释 + Claude Code 生成 README
```

## 5.3 AI 辅助需求分析：从模糊想法到用户故事

### Prompt 模板：需求澄清

当你有一个模糊的想法时，比如"我想做一个 AI 研究平台"，不要直接开始写代码。先让 AI 帮你理清需求：

```
我想做一个"AI 深度研究与报告生成平台"。用户输入研究主题，系统自动检索资料、分析观点、生成报告。

请帮我：
1. 列出这个系统的核心用户角色
2. 为每个角色定义 3-5 个核心用户故事（格式：作为[角色]，我想要[功能]，以便[价值]）
3. 识别出技术风险最高的 3 个功能点
4. 建议一个最小可行产品（MVP）的功能范围
```

Cursor 或 Claude Code 会给你一个结构化的输出，你可以在此基础上迭代。

### 迭代技巧

不要一次要太多。先让 AI 生成用户故事，然后挑选一个最有价值的，让 AI 继续细化：

```
"作为研究员，我想要输入研究主题并生成深度报告"这个故事太大了。

请把它拆分成 3 个可交付的迭代，每个迭代都有明确的价值点和验收标准。
```

## 5.4 AI 辅助架构设计：技术选型与边界梳理

### 用 AI 做技术选型讨论

当你面临技术选型时（比如"选 Redis 还是 PostgreSQL 做缓存"），可以让 AI 扮演架构评审员：

```
我在设计一个"AI 深度研究平台"的缓存策略。

候选方案：
A. PostgreSQL + 应用层内存缓存
B. Redis 独立缓存层
C. PostgreSQL + Redis 双层缓存

业务特点：
- 报告生成结果需要缓存（TTL 24 小时）
- 用户会话数据需要快速读取
- AI API 调用结果需要缓存（按请求参数）
- 预算有限，不想维护太多组件

请对比三个方案的优缺点，给出推荐方案，并说明理由。
```

AI 不会替你拍板，但它能帮你结构化地梳理利弊，避免遗漏关键考量。

### 架构图生成

AI 不太擅长画复杂的架构图，但可以用 Mermaid 或 ASCII 生成简单的结构图：

```
请用 Mermaid flowchart 画一个 Clean Architecture 简化版的架构图。
包含：Handler（HTTP 接口层）、Service（业务逻辑层）、Repository（数据访问层）、Domain（领域模型层）。
用中文标注，显示依赖方向（外层依赖内层）。
```

Cursor 会生成 Mermaid 代码，你可以直接粘贴到 Markdown 中渲染。

## 5.5 AI 辅助原型设计：从文字到可运行代码

### 从描述到线框图

最快的原型方式不是画 Figma，而是让 AI 直接生成可运行的 HTML：

```
请生成一个 React 页面，用于"创建研究报告"。

页面元素：
- 顶部导航栏：Logo、用户头像下拉菜单
- 左侧边栏：研究任务列表
- 中间主区域：
  - 标题："创建新研究"
  - 输入框：研究主题（占位符："例如：2024 年固态电池技术进展"）
  - 下拉选择：研究深度（快速/标准/深度）
  - 滑块：目标报告长度（1000-10000 字）
  - 多选：重点领域（技术/市场/政策/竞争）
  - 按钮："开始研究"（主按钮，蓝色）

使用 Tailwind CSS 样式，Shadcn UI 组件风格。
```

Cursor 会生成完整的 React 组件代码，你可以在 Vite 项目里直接运行看效果。

### 从线框到高保真

当你有了基本结构后，可以让 AI 细化：

```
给上面的页面加上：
1. 表单校验（主题不能为空，长度必须在范围内）
2. 加载状态（点击"开始研究"后显示进度条和步骤说明）
3. 响应式布局（移动端侧边栏变成抽屉）
4. 暗色模式支持
```

### 原型验证

让 AI 检查你的原型是否遗漏了关键场景：

```
请审查上面的"创建研究"页面，检查是否遗漏了以下场景：
1. 网络错误时的处理
2. 用户未登录时的状态
3. 表单重置功能
4. 研究主题历史记录
5. 预算/Token 消耗提示

对每个遗漏的场景，给出建议的 UI 处理方案。
```

## 5.6 AI 辅助编码：Prompt 怎么写最有效

### 好的 Prompt 结构

向 AI 描述编码任务时，最有效的结构是：**背景 + 目标 + 约束 + 示例**。

**反例**（太模糊）：
```
写一个用户登录接口
```

**正例**（结构化）：
```
背景：我们在用 Go + Gin 开发 RESTful API，采用 Clean Architecture 三层结构（Handler → Service → Repository）。

目标：实现用户登录接口 POST /api/auth/login

约束：
- 接收 JSON：{ "email": string, "password": string }
- 验证邮箱格式和密码长度（不少于 8 位）
- 成功后返回 JWT Access Token 和 Refresh Token
- 错误时返回统一格式的 JSON：{ "code": 400/401, "message": string }
- Handler 层只做参数绑定和响应构造，业务逻辑在 Service 层
- 使用项目已有的 pkg/errors 和 pkg/response 包

示例（项目已有代码风格）：
```go
// handler/report_handler.go
func (h *ReportHandler) Create(c *gin.Context) {
    var req CreateRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
        return
    }
    // ...
}
```

请生成完整的 Handler 代码和对应的 Service 接口定义。
```

### 上下文引用技巧

Cursor 的 `@` 符号是核心竞争力：

- `@file:report_handler.go` —— 引用某个文件，让 AI 了解项目风格
- `@func:Create` —— 引用某个函数，让 AI 保持一致的实现模式
- `@dir:internal/handler` —— 引用整个目录，让 AI 了解整体架构

Claude Code 的上下文管理：

```bash
# 把文件加入上下文
/add internal/handler/report_handler.go

# 查看当前上下文
/ls

# 清空上下文
/clear
```

### 渐进式生成

不要一次让 AI 生成 500 行代码。分步骤来：

1. **先生接口**：让 AI 定义 Service 接口和 DTO
2. **再实现核心逻辑**：让 AI 实现 Service 层的核心业务
3. **最后补边缘情况**：错误处理、日志、边界条件

每步都编译验证，有问题立刻反馈给 AI 修正。

## 5.7 AI 辅助调试：错误分析、日志解读、根因定位

### 粘贴错误让 AI 分析

遇到报错时，把完整错误信息（包括堆栈）粘贴给 AI：

```
运行测试时报错：

=== RUN   TestReportService_Create
    report_service_test.go:23: 创建报告失败: sql: Scan error on column index 5, name "sources": unsupported Scan, storing driver.Value type string into type *[]sql.NullString

请分析：
1. 这个错误的根本原因是什么？
2. 在 scanReport 函数中应该怎么修改？
3. 为什么 pgx/stdlib 返回的 text[] 不能直接扫描到 []sql.NullString？
```

### 日志解读

生产环境的日志往往是海量的。让 AI 帮你找异常模式：

```
以下是我们后端服务过去一小时的错误日志片段（已脱敏）：

[ERROR] 2024-01-15T10:23:45Z handler=report status=500 error="pq: remaining connection slots are reserved for non-replication superuser connections"
[ERROR] 2024-01-15T10:23:47Z handler=report status=500 error="pq: remaining connection slots are reserved for non-replication superuser connections"
[ERROR] 2024-01-15T10:24:01Z handler=health status=500 error="context deadline exceeded"

请分析：
1. 最可能的原因是什么？
2. 我们应该检查哪些配置？
3. 临时缓解方案和长期解决方案分别是什么？
```

### 让 AI 生成调试代码

```
请给 ReportService.Create 方法加上详细的调试日志，要求：
1. 使用标准库 log，不需要额外依赖
2. 记录输入参数（脱敏处理密码等敏感字段）
3. 记录每个关键步骤的耗时
4. 错误时记录完整的错误链
5. 只在 debug 模式下输出
```

## 5.8 AI 辅助重构：识别坏味道、提取函数、优化并发

### 让 AI 识别代码坏味道

把一段代码贴给 AI，让它指出问题：

```
请审查以下代码，指出其中的"代码坏味道"，并给出重构建议：

```go
func (s *reportService) ListByUser(ctx context.Context, userID int64, page, pageSize int) ([]*domain.Report, int, error) {
    if page < 1 { page = 1 }
    if pageSize < 1 { pageSize = 20 }
    if pageSize > 100 { pageSize = 100 }

    all, err := s.repo.ListByUser(ctx, userID, 10000, 0)
    if err != nil {
        return nil, 0, errors.NewInternal("查询报告列表失败", err)
    }
    total := len(all)

    offset := (page - 1) * pageSize
    reports, err := s.repo.ListByUser(ctx, userID, pageSize, offset)
    if err != nil {
        return nil, 0, errors.NewInternal("查询报告列表失败", err)
    }

    return reports, total, nil
}
```

要求按优先级排序，每个问题给出：问题描述、为什么是坏味道、重构后的代码。
```

### 批量重构

Claude Code 在批量重构上特别强。比如你要给所有 Handler 加上统一错误处理：

```bash
$ claude
> 给 internal/handler/ 下所有 Handler 方法的错误处理改成使用 pkg/response 包。
> 具体模式：
>   - 如果是 domain.NotFoundError，返回 response.NotFound
>   - 如果是 validation error，返回 response.BadRequest  
>   - 其他返回 response.InternalServerError
> 请逐个文件修改，修改后运行 go build ./... 验证编译通过。
```

## 5.9 AI 辅助测试：生成用例、边界条件、覆盖率补充

### 生成测试用例

给 AI 一个函数，让它生成全面的测试：

```
请为以下函数生成完整的 Go 单元测试，使用表格驱动测试风格：

```go
func (s *reportService) Cancel(ctx context.Context, id int64) error {
    report, err := s.repo.GetByID(ctx, id)
    if err != nil {
        return errors.NewNotFound("report", id)
    }
    if !report.CanCancel() {
        return errors.NewValidation("status", fmt.Sprintf("当前状态 %s 不允许取消", report.Status))
    }
    if err := s.repo.UpdateStatus(ctx, id, domain.ReportStatusFailed); err != nil {
        return errors.NewInternal("取消报告失败", err)
    }
    return nil
}
```

要求：
1. 覆盖正常流程和所有错误分支
2. 使用 mock repository（假设已有 memory.NewReportRepository）
3. 包含边界条件：id=0、已完成状态的报告、不存在的报告
4. 测试函数名使用 TestReportService_Cancel_xxx 格式
```

### 补充边界条件

AI 擅长找出你没想到的边界：

```
我已经为 Create 方法写了这些测试用例：空标题、空主题、正常创建。

请帮我补充更多边界条件测试：
1. 标题超长（超过 200 字符）
2. 特殊字符和 SQL 注入尝试
3. 并发创建（竞态条件）
4. context 超时
```

## 5.10 AI 辅助代码审查：静态分析之外的智能检查

### 让 AI 扮演 Reviewer

在提交 PR 前，让 Claude Code 先做一轮审查：

```bash
$ claude
> 请审查当前工作目录的所有变更，关注以下方面：
> 1. 是否有明显的逻辑错误
> 2. 错误处理是否完整
> 3. 是否有资源泄漏（未关闭的 rows、未释放的连接）
> 4. 并发安全性
> 5. 是否符合项目的 Clean Architecture 分层规范
>
> 对每个发现的问题，标注严重级别（严重/建议/可选），并给出修改建议。
```

### 生成 Review 评论

如果你是 reviewer，可以用 AI 帮你写评论：

```
请帮我写一段代码审查评论，针对以下代码：

```go
func GetReport(c *gin.Context) {
    id := c.Param("id")
    report, _ := db.Query("SELECT * FROM reports WHERE id = " + id)
    c.JSON(200, report)
}
```

要求：
1. 指出所有安全问题
2. 给出修改后的代码
3. 语气专业但友善
```

## 5.11 AI 辅助文档：注释、README、API 文档

### 生成函数注释

Copilot 和 Cursor 都能自动生成符合 Go 规范的注释：

```go
// Create 创建新的研究报告
// 参数校验失败时返回 KindValidation 错误
func (s *reportService) Create(ctx context.Context, userID int64, title, topic string) (*domain.Report, error) {
```

但 AI 生成的注释往往太泛化。更好的 Prompt：

```
请为以下函数生成注释，要求：
1. 说明函数的副作用（比如修改了哪些状态）
2. 说明所有可能返回的错误类型
3. 说明调用方需要满足的前置条件
4. 不要重复函数名中已经 obvious 的信息
```

### 生成 README 更新

当你新增了一个模块后，让 AI 帮你更新 README：

```
我在项目中新增了以下功能，请帮我更新 README.md 的"快速开始"和"项目结构"部分：

新增：
1. Swagger API 文档（访问 /swagger）
2. 数据库迁移工具（make migrate）
3. PostgreSQL Repository 实现

README 当前内容：[粘贴当前 README]
```

### 生成 API 文档

## 5.12 上下文管理艺术：怎么给 AI 喂代码才能少幻觉

AI 编程助手最大的问题是**幻觉**——它会自信地编造不存在的函数、错误的参数、甚至虚构的库。减少幻觉的关键是**上下文管理**。

### 规则 1：给足上下文，但不要过载

Cursor 的上下文窗口大约是 200K token，Claude Code 也是类似。但填得太满会导致：
- 响应变慢
- AI 注意力分散
- 容易忽略关键信息

**最佳实践**：
- 一次只给 AI 3-5 个相关文件
- 大文件只给相关函数（用 `@func:Name`）
- 用 `.cursorignore` 或 `.claudeignore` 排除无关文件（node_modules、vendor、日志等）

### 规则 2：显式引用 vs 隐式引用

**隐式引用**（不好的）：
```
给这个函数加上错误处理（AI 不知道"错误处理"在项目里是什么模式）
```

**显式引用**（好的）：
```
参考 @file:pkg/errors/errors.go 中的错误类型，给这个函数加上统一的错误处理
```

### 规则 3：迭代式对话

不要期望一次 Prompt 就得到完美结果。分轮次来：

1. 第一轮：生成基础实现
2. 第二轮：补充错误处理
3. 第三轮：补充日志和注释
4. 第四轮：运行测试，修复问题

每轮都基于上一轮的结果，AI 会越改越准。

### 规则 4：验证一切

**AI 写的代码必须编译、必须测试通过，才能合并**。这是铁律。

```bash
# AI 生成代码后，立刻验证
$ go build ./...
$ go test ./...
$ golangci-lint run ./...
```

如果报错，把错误信息贴回给 AI，让它修正。通常 2-3 轮就能稳定。

## 5.13 AI 辅助开发的陷阱与底线

### 陷阱 1：幻觉代码

AI 可能会生成调用不存在函数的代码：

```go
// AI 生成的代码
result, err := db.ExecContext(ctx, "INSERT INTO ...")
report.ID = result.LastInsertId()  // PostgreSQL 不支持！
```

PostgreSQL 的 `RETURNING id` 才是正确做法。AI 因为训练数据里 MySQL 更多，可能默认用 `LastInsertId`。

**应对**：对数据库操作、框架特定的 API，必须人工核对文档。

### 陷阱 2：安全漏洞

AI 对安全问题的敏感度不够。它可能生成：

```go
// 危险的代码
query := fmt.Sprintf("SELECT * FROM reports WHERE title = '%s'", title)
```

或者生成硬编码的密钥、不安全的随机数、不完整的输入校验。

**应对**：所有涉及安全相关的代码（认证、授权、输入处理）必须经过人工审查。用 `gosec` 等工具做自动化安全检查。

### 陷阱 3：版权风险

AI 训练数据包含大量开源代码。它可能生成与某个开源项目过于相似的代码片段，带来许可证兼容性问题。

**应对**：
- 对 AI 生成的代码进行差异化修改（重命名、重构结构）
- 企业项目建议使用有版权保障承诺的 AI 工具（如 GitHub Copilot 的商业版）
- 核心算法和业务逻辑不要完全依赖 AI 生成

### 陷阱 4：过度依赖

最危险的陷阱：**习惯了 AI 补全后，失去了独立解决问题的能力**。

当你遇到 AI 无法解决的问题时（比如一个罕见的框架 bug），如果你已经丧失了读文档、调试、搜索的能力，就会束手无策。

**应对**：
- 定期"裸写"代码：关闭 AI 助手，手写一个完整功能
- 理解 AI 生成的每一行代码，不要无脑接受
- 保持阅读官方文档的习惯

### 底线原则

| 场景 | AI 可以做什么 | AI 不能做什么 | 必须由人把关 |
|------|-------------|-------------|-----------|
| 生成 boilerplate | 完全信任 | — | 编译检查 |
| 写业务逻辑 | 生成初稿 | 直接合并 | 逻辑正确性 |
| 写测试 | 生成用例 | 替代思考边界条件 | 覆盖率、边界条件 |
| 写 SQL | 生成查询 | 直接用于生产 | 注入检查、性能 |
| 写配置/密钥 | — | 任何情况下都不行 | 安全审查 |
| 代码审查 | 初步扫描 | 替代人工 review | 架构设计、安全 |

## 本章小结

1. **工具选型**：Copilot 日常补全 + Cursor 功能开发 + Claude Code 批量重构，是三件套最优组合。
2. **Prompt 工程**：背景 + 目标 + 约束 + 示例的结构化 Prompt，比模糊描述效果好 10 倍。
3. **上下文管理**：一次 3-5 个文件，显式引用，迭代式对话，避免幻觉。
4. **渐进式生成**：先接口、再核心、再边缘，每步都编译验证。
5. **安全底线**：涉及安全、密钥、SQL 的代码必须人工审查，AI 只是辅助。
6. **防过度依赖**：定期裸写代码，保持独立解决问题的能力。
