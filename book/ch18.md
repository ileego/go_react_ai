# 第18章 Tailwind CSS 与设计系统构建

第17章我们把前端从单页扩展成了多页应用：路由配置、嵌套布局、路由参数、导航守卫、懒加载都已就位。但回头看代码会发现一个尴尬的事实：`Root`、`ReportCenterLayout`、`LoginPage` 还在用内联 `style={{ ... }}` 写样式；`Button` 组件虽然用了 Tailwind 类名，但项目里压根没安装 Tailwind，那些类名只是"字符串装饰"。

本章学习目标：

- 理解 Utility-First 设计哲学，能权衡它与手写 CSS / CSS-in-JS 的优劣；
- 掌握 Tailwind v3 的核心配置：`tailwind.config` / `postcss.config` / `@tailwind` 指令；

- 熟练使用颜色、文字、边框、间距、过渡、变换等常用样式工具类；

- 理解 JIT 模式与 `content` 扫描机制，验证生产 CSS 体积。

## 18.1 Utility-First 设计哲学：效率与一致性的平衡

### 18.1.1 从"写样式"到"拼类名"

传统 CSS 的工作流是：先给元素起类名，再去 CSS 文件里写规则。

```css
.card {
  padding: 1rem;
  border-radius: 0.5rem;
  background-color: #ffffff;
  box-shadow: 0 4px 6px -1px rgb(0 0 0 / 0.1);
}
```

```html
<div class="card">...</div>
```

这个流程的问题在于：每增加一个组件，就要发明一个新类名、写一组规则，团队里很快会出现 `card`、`card-v2`、`card-new`、`report-card` 等命名灾难。而一旦要统一调整圆角或阴影，你可能要改十几个 CSS 文件。

Tailwind 把粒度拆到最小：每个 CSS 属性对应一个类名。你不再"写样式"，而是"拼类名"。

```html
<div class="rounded-lg bg-white p-4 shadow-md">...</div>
```

这听起来像是"反语义化"，但关键在于：**语义被提升到了配置层**。`rounded-lg`、`p-4`、`shadow-md` 不是随意值，它们都来自 `tailwind.config.js` 里定义的设计令牌。团队里所有人用的都是同一套刻度，一致性自然得到保证。

### 18.1.2 设计令牌是唯一真相源

一个设计系统的核心不是"有多少组件"，而是**设计令牌（Design Tokens）**：颜色、字体、间距、圆角、阴影、动画时长等基础决策。Tailwind 的配置文件就是这个真相源。

```mermaid
graph TD
    A[设计令牌<br/>tailwind.config] --> B[Tailwind 工具类<br/>bg-brand-600 p-4 shadow-card]
    B --> C[可复用组件<br/>Button / Card / Input]
    C --> D[页面布局<br/>Root / ReportCenterLayout]
    D --> E[业务页面<br/>Home / Reports / Login]
```

在这个层级里：

- 设计令牌变化时，所有使用该令牌的组件和页面同步更新；
- 组件层只做"哪些令牌组合在一起"的决策；
- 页面层专注于业务逻辑，而不是纠结像素值。

### 18.1.3 Tailwind 与其他方案对比

| 方案         | 优点                                   | 缺点                         | 适用场景                          |
| ------------ | -------------------------------------- | ---------------------------- | --------------------------------- |
| 内联 `style` | 零依赖、直观                           | 难以复用、无设计约束         | 教学示例、临时调试                |
| CSS Modules  | 局部作用域、无命名冲突                 | 样板文件多、命名开销大       | 大型遗留项目、需要严格隔离        |
| CSS-in-JS    | 动态主题强、组件即样式                 | 运行时开销、包体积增加       | 高度动态主题、复杂条件样式        |
| Tailwind CSS | 配置即规范、JIT 按需生成、无运行时开销 | 需要记忆类名、类名字符串较长 | 现代 React/Vue 项目、设计系统驱动 |

> **注意**：Utility-First 不是"不要语义化"。语义在 `tailwind.config` 和组件命名里；HTML 上的类名负责表达"长什么样"，而不是"是什么业务概念"。

## 18.2 Tailwind 核心：配置、插件、自定义工具类

### 18.2.1 安装 Tailwind CSS v3

本书前端使用 Vite + React 19 + TypeScript。Tailwind v3 与这套栈配合最成熟，且与目录中"JIT 模式 / purge 策略"的概念一致。执行以下命令：

```bash
cd src/frontend
pnpm add -D tailwindcss@^3.4.17 postcss autoprefixer
```

> **注意**：仓库 `package.json` 设置了 `"type": "module"`，因此配置文件统一用 `.cjs` 扩展名，避免 `module.exports` 在 ESM 环境下报错。

### 18.2.2 三件套：tailwind.config.cjs / postcss.config.cjs / index.css

### 18.2.3 `content` 字段：Tailwind 的 purge 机制

Tailwind v3 默认启用 JIT（Just-In-Time）模式。它不会把所有可能的类都生成出来，而是扫描 `content` 里指定的文件，只生成其中**实际出现过的类**。

```cjs
content: ['./index.html', './src/**/*.{js,ts,jsx,tsx}'],
```

这就是 v3 中的 "purge"：

- 开发时，只要你在 JSX/TSX 里写一个类名，Tailwind 立刻生成对应的 CSS；
- 生产构建时，未使用的类不会进入最终包，CSS 体积极小；
- 如果 `content` 路径写漏了（比如忘记包含 `.tsx`），对应的类就会失效。

> **注意**：不要把整个 `node_modules` 写进 `content`。范围太大会拖慢扫描速度，而且大部分库都自带样式，不需要 Tailwind 处理。

### 18.2.4 自定义插件与 `@apply`

如果某些样式反复出现，可以用 Tailwind 插件把它们注册成新的工具类。上面的配置里就添加了一个 `.text-balance` 工具类：

```cjs
plugins: [
  require('tailwindcss/plugin')(({ addUtilities }) => {
    addUtilities({
      '.text-balance': { textWrap: 'balance' },
    })
  }),
],
```

`@apply` 则用于在全局 CSS 里复用工具类，比如打印样式或全局动画：

```css
@layer components {
  .prose p {
    @apply mb-4 leading-relaxed;
  }
}
```

> **提示**：不要把每个组件都包成一个 `@apply` 自定义类。那样会重新发明"语义化类名"，失去 Utility-First 的优势。`@apply` 适合极少数需要跨组件复用且不适合做成组件的场景。

## 18.3 布局系统：Flexbox、Grid、响应式断点

### 18.3.1 常用布局工具类

Tailwind 把 Flexbox 和 Grid 的常用属性都映射成了类名，写布局时不再需要写大量 CSS。

| 能力      | 类名示例                                              | 说明                                                           |
| --------- | ----------------------------------------------------- | -------------------------------------------------------------- |
| 开启 Flex | `flex`                                                | 将元素设为 Flex 容器，子元素默认在一行排列                   |
| 方向      | `flex-row`、`flex-col`、`flex-row-reverse`            | 主轴方向：水平、垂直、水平反向                               |
| 对齐      | `items-center`、`justify-between`、`justify-end`      | `items-*` 控制交叉轴对齐；`justify-*` 控制主轴对齐           |
| 自动边距  | `mx-auto`、`ml-auto`                                  | 水平居中或把元素推到右侧；`auto` 会吃掉剩余空间               |
| 开启 Grid | `grid`                                                | 将元素设为 Grid 容器                                         |
| 列数      | `grid-cols-2`、`grid-cols-3`、`grid-cols-[200px_1fr]` | 定义列数；方括号为任意值，适合固定 + 自适应组合             |
| 间距      | `gap-4`、`gap-x-2`、`space-y-2`                       | `gap-*` 用于 Flex/Grid 子元素间隙；`space-*` 用于普通块级容器 |
| 自适应    | `grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`           | 小屏 1 列，大手机 2 列，桌面 3 列，渐进响应                  |

> **提示**：`grid-cols-[200px_1fr]` 中的方括号是 Tailwind 的任意值语法，表示"第一列固定 200px，第二列自适应"。18.3.2 节会详细解释它的工作原理与记忆方法。

使用这些工具类时，最容易混淆的是两组概念：

- **`items-*` vs `justify-*`**：`items-*` 控制交叉轴对齐（例如 `items-center` 让 Flex 子元素在垂直方向居中）；`justify-*` 控制主轴对齐（例如 `justify-between` 让子元素两端对齐）。
- **`gap-*` vs `space-*`**：`gap-*` 必须用在 Flex 或 Grid 容器上；`space-y-*` / `space-x-*` 则用于普通块级容器，通过给非首个子元素加外边距实现间距。现代写法推荐优先使用 `gap-*`。
- **响应式前缀**：`grid-cols-1 sm:grid-cols-2 lg:grid-cols-3` 表示"小屏 1 列，大手机 2 列，桌面 3 列"，这是移动优先的渐进式写法，18.3.2 会详细展开。

### 18.3.2 响应式断点

Tailwind 采用**移动优先（Mobile-First）**策略：默认样式对应最小屏幕，再用 `sm:`、`md:`、`lg:` 等前缀覆盖更大屏幕。

| 断点  | 最小宽度 | 典型设备 |
| ----- | -------- | -------- |
| `sm`  | 640px    | 大手机   |
| `md`  | 768px    | 平板     |
| `lg`  | 1024px   | 小桌面   |
| `xl`  | 1280px   | 桌面     |
| `2xl` | 1536px   | 大屏     |

```mermaid
flowchart LR
    A[默认样式<br/>移动优先] --> B[sm: 640px]
    B --> C[md: 768px]
    C --> D[lg: 1024px]
    D --> E[xl: 1280px]
```

例如 `grid-cols-1 md:grid-cols-[200px_1fr]` 表示：小屏幕单列，平板及以上变为"200px 侧边栏 + 自适应内容区"。

#### 为什么写成 `md:grid-cols-[200px_1fr]`？

这个类名拆开来看：

- `md:`：在 `768px` 及以上生效的响应式前缀；
- `grid-cols-`：对应 CSS 属性 `grid-template-columns`；
- `[200px_1fr]`：Tailwind 的**任意值语法**，方括号里的内容会直接变成 CSS 值。

Tailwind 会把下划线 `_` 替换成空格，因此 `[200px_1fr]` 实际生成的 CSS 是：

```css
grid-template-columns: 200px 1fr;
```

这表示 Grid 容器有两列：第一列**固定 200 像素**，第二列**占据剩余全部可用空间**（`1fr` = 1 fraction，即一份剩余空间）。把它和响应式前缀组合，就得到了"小屏单列、中等屏幕以上左侧固定侧边栏 + 右侧自适应内容"的经典后台布局。

#### 为什么要用任意值，而不用预设类？

Tailwind 预设了 `grid-cols-1`、`grid-cols-2`、`grid-cols-3` 等常用列数，但它们都是**等宽列**。如果你需要"固定宽度 + 自适应"这种非等宽组合，预设类无法表达，就必须借助任意值。

常见场景：

| 需求                          | 类名                              | 生成的 `grid-template-columns` |
| ----------------------------- | --------------------------------- | ------------------------------ |
| 左侧固定 200px，右侧自适应    | `grid-cols-[200px_1fr]`           | `200px 1fr`                    |
| 左侧自适应，右侧固定 300px    | `grid-cols-[1fr_300px]`           | `1fr 300px`                    |
| 左侧 250px，中间自适应，右侧 250px | `grid-cols-[250px_1fr_250px]` | `250px 1fr 250px`              |
| 三列按比例 1:2:1              | `grid-cols-[1fr_2fr_1fr]`         | `1fr 2fr 1fr`                  |
| 固定像素与比例混合            | `grid-cols-[120px_1fr_2fr]`       | `120px 1fr 2fr`                |

#### 怎么记？

记忆这套语法可以抓住三个要点：

1. **方括号就是"直接写 CSS 值"**：`[200px_1fr]` 里的内容几乎和 CSS 值一模一样，只是用下划线代替空格；
2. **`fr` 是 fraction（份数）**：`1fr` 占一份剩余空间，`2fr` 占两份，没有 `fr` 的固定值会先被满足；
3. **从左到右依次对应列**：`200px_1fr` 就是第一列 200px，第二列自适应。

#### 任意值语法也适用于其他属性

方括号任意值不只在 Grid 里用，几乎所有 Tailwind 工具类都支持：

```html
<!-- 宽度任意值 -->
<div class="w-[123px]"></div>

<!-- 背景色任意值 -->
<div class="bg-[#1da1f2]"></div>

<!-- 定位任意值 -->
<div class="top-[calc(100%-2rem)]"></div>

<!-- 最大宽度任意值 -->
<div class="max-w-[1200px]"></div>
```

> **注意**：任意值虽然灵活，但会绕过设计令牌。如果某个值反复出现（比如侧边栏固定 200px），建议把它抽成配置：`spacing: { sidebar: '200px' }`，然后使用 `grid-cols-[theme(spacing.sidebar)_1fr]`，或直接用 CSS 变量。这样既能保持灵活性，又不破坏一致性。

### 18.3.3 容器、定位与层级控制

除了 Flex 与 Grid，页面还需要控制元素的最大宽度、绝对/相对定位、层叠顺序和溢出行为。Tailwind 把这些常用样式也映射成了工具类。

| 能力       | 类名示例                                                       | 说明                                           |
| ---------- | -------------------------------------------------------------- | ---------------------------------------------- |
| 最大宽度   | `max-w-md`、`max-w-3xl`、`max-w-full`、`max-w-[1200px]`        | 限制内容宽度，`[1200px]` 为方括号任意值；配合 `mx-auto` 居中 |
| 容器       | `container`、`mx-auto`                                         | `container` 默认带响应式最大宽度               |
| 定位上下文 | `relative`、`absolute`、`fixed`、`sticky`                      | 开启对应定位模式                               |
| 定位偏移   | `top-4`、`right-0`、`bottom-1/2`、`left-1/2`                   | 配合 `absolute` / `fixed` 使用                 |
| 层级       | `z-0`、`z-10`、`z-50`                                          | 控制层叠顺序，模态框通常用 `z-50`              |
| 溢出       | `overflow-hidden`、`overflow-auto`、`overflow-x-auto`          | 裁剪或滚动超出内容                             |
| 显示/隐藏  | `block`、`hidden`、`invisible`、`opacity-0`                    | 控制元素可见性与盒模型                         |

一个常见的"顶部固定导航 + 居中内容"组合：

```html
<body class="bg-white dark:bg-slate-900">
  <header class="fixed inset-x-0 top-0 z-50 border-b border-gray-200 bg-white/80 backdrop-blur dark:border-slate-700 dark:bg-slate-900/80">
    ...
  </header>

  <main class="mx-auto mt-16 max-w-5xl px-4 py-6">
    ...
  </main>
</body>
```

要点：

- `fixed inset-x-0 top-0` 把元素固定在视口顶部，左右撑满；
- `z-50` 确保导航浮在主要内容之上；
- `bg-white/80 backdrop-blur` 使用 Tailwind 的背景透明度与毛玻璃工具类；
- `mx-auto max-w-5xl px-4` 把主内容限制在舒适的阅读宽度并自动居中。

### 18.3.4 项目实战：重写报告中心布局

第17章的 `ReportCenterLayout` 用内联 Grid 写死了侧边栏。现在用 Tailwind 重写，并加入响应式能力：

```tsx
// 文件: src/frontend/src/features/reports/pages/ReportCenterLayout.tsx（教学示例）

import { Outlet, Link } from 'react-router-dom'

export function ReportCenterLayout() {
  return (
    <div className="grid grid-cols-1 gap-6 md:grid-cols-[200px_1fr]">
      <aside className="border-b border-gray-200 pb-4 md:border-b-0 md:border-r md:pr-4 dark:border-slate-700">
        <nav className="flex flex-row gap-3 md:flex-col">
          <Link to="/reports" className="hover:text-brand-600 dark:hover:text-brand-400">
            全部报告
          </Link>
          <Link to="/reports/new" className="hover:text-brand-600 dark:hover:text-brand-400">
            新建报告
          </Link>
        </nav>
      </aside>

      <section>
        <Outlet />
      </section>
    </div>
  )
}
```

改造点：

- `grid-cols-1 md:grid-cols-[200px_1fr]` 实现响应式两栏；
- 小屏幕下侧边栏在内容上方、横向排列；中等屏幕以上恢复左侧竖向导航；
- 用 `dark:border-slate-700` 预留暗色模式样式。

```tsx
        <h3 className="mb-2 font-medium">响应式 Grid 布局</h3>

  )
```

## 18.4 设计令牌：颜色、字体、间距、圆角、阴影

### 18.4.1 为什么需要设计令牌

没有令牌时，一个"品牌蓝"可能在项目里出现 `#2563eb`、`#3b82f6`、`blue-600` 等多种写法。设计令牌把这些决策收敛到一个地方：

```cjs
// tailwind.config.cjs 片段

colors: {
  brand: {
    50: '#eff6ff',
    100: '#dbeafe',
    // ...
    600: '#2563eb',
    // ...
  },
}
```

之后团队里只使用 `bg-brand-600`、`text-brand-700`、`ring-brand-300`。品牌色调整时，改一处即可全局生效。

### 18.4.2 扩展默认主题而非覆盖

推荐用 `theme.extend` 而不是直接覆盖 `theme`。这样可以保留 Tailwind 的 `gray`、`red`、`blue` 等完整色板，降低迁移成本。

我们已经配置的令牌包括：

- `colors.brand.*`：品牌主色；
- `fontFamily.sans`：全局无衬线字体栈；
- `spacing.18`：自定义 `4.5rem` 间距；
- `borderRadius.2xl`：`1rem` 大圆角；
- `boxShadow.card`：卡片阴影；
- `animation.fadeIn`：页面淡入动画。

### 18.4.3 项目实战：用品牌色改造 Button

```tsx
// 文件: src/frontend/src/shared/components/Button.tsx（教学示例）

import type { ButtonHTMLAttributes, ReactNode } from 'react'

interface Props extends ButtonHTMLAttributes<HTMLButtonElement> {
  children: ReactNode
  variant?: 'primary' | 'secondary' | 'outline' | 'danger'
}

export function Button({ children, variant = 'primary', ...rest }: Props) {
  const base =
    'px-4 py-2 rounded font-medium transition-colors focus:outline-none focus:ring-2 focus:ring-offset-2 disabled:opacity-50 disabled:cursor-not-allowed'
  const styles = {
    primary:
      'bg-brand-600 text-white hover:bg-brand-700 focus:ring-brand-300 focus:ring-offset-white dark:focus:ring-offset-slate-900',
    secondary:
      'bg-gray-200 text-gray-800 hover:bg-gray-300 focus:ring-gray-300 focus:ring-offset-white dark:bg-slate-700 dark:text-slate-100 dark:hover:bg-slate-600 dark:focus:ring-offset-slate-900',
    outline:
      'border-2 border-gray-300 text-gray-700 hover:bg-gray-100 focus:ring-gray-300 focus:ring-offset-white dark:border-slate-600 dark:text-slate-200 dark:hover:bg-slate-800 dark:focus:ring-offset-slate-900',
    danger:
      'bg-red-600 text-white hover:bg-red-700 focus:ring-red-300 focus:ring-offset-white dark:focus:ring-offset-slate-900',
  }

  return (
    <button className={`${base} ${styles[variant]}`} {...rest}>

    </button>
  )
}
```

### 18.4.6 颜色、文字、边框与背景工具类详解

Tailwind 把视觉样式拆成细粒度工具类，掌握它们之间的组合方式，就能在不写一行 CSS 的情况下完成绝大多数界面。

#### 颜色系统

Tailwind 默认提供 `gray`、`red`、`orange`、`yellow`、`green`、`blue`、`purple`、`pink` 等完整色板，每种颜色从 `50` 到 `950` 共 11 个阶。品牌色通过 `theme.extend.colors` 注入后，用法与默认色完全一致。

| 用途       | 类名示例                                              | 生成的 CSS 属性          |
| ---------- | ----------------------------------------------------- | ------------------------ |
| 文字色     | `text-brand-600`、`text-gray-900`                     | `color`                  |
| 背景色     | `bg-brand-100`、`bg-white`、`bg-slate-800`            | `background-color`       |
| 边框色     | `border-gray-200`、`border-brand-600`                 | `border-color`           |
| 轮廓环     | `ring-brand-300`、`ring-2`                            | `--tw-ring-color` / 宽度 |
| 分隔线     | `divide-gray-200`                                     | 子元素 `border-color`    |

暗色模式下，通常把背景从 `bg-white` 切换为 `bg-slate-900`，文字从 `text-gray-900` 切换为 `text-slate-100`，中间层级用 `text-gray-300` / `text-slate-300` 保持可读性。这些配对应写进项目规范，而不是每次临场发挥。

#### 文字样式

| 能力     | 类名示例                                         | 说明                           |
| -------- | ------------------------------------------------ | ------------------------------ |
| 字号     | `text-sm`、`text-base`、`text-lg`、`text-2xl`    | 默认以 `rem` 为单位的 9 档字号 |
| 字重     | `font-normal`、`font-medium`、`font-semibold`、`font-bold` | 400 / 500 / 600 / 700          |
| 行高     | `leading-none`、`leading-tight`、`leading-relaxed`、`leading-loose` | 紧凑到宽松                     |
| 字间距   | `tracking-tight`、`tracking-wide`                | 字母间距                       |
| 对齐     | `text-left`、`text-center`、`text-right`、`text-justify` | 水平对齐                       |
| 装饰     | `underline`、`line-through`、`no-underline`      | 下划线与删除线                 |
| 转换     | `uppercase`、`lowercase`、`capitalize`           | 大小写转换                     |
| 换行     | `truncate`、`text-ellipsis`、`break-words`       | 单行截断与长单词换行           |

> **提示**：本项目的 `text-balance` 自定义工具类通过插件注入，用于标题的两端平衡排版。

#### 边框与背景

| 能力     | 类名示例                                                   | 说明                           |
| -------- | ---------------------------------------------------------- | ------------------------------ |
| 边框宽度 | `border`、`border-2`、`border-t`、`border-x`               | 四边或单边边框                 |
| 圆角     | `rounded`、`rounded-lg`、`rounded-2xl`、`rounded-full`     | 统一圆角，或 `rounded-t` 单边  |
| 背景图   | `bg-gradient-to-r from-brand-500 to-purple-600`            | 线性渐变                       |
| 背景尺寸 | `bg-cover`、`bg-contain`、`bg-center`                      | 背景图定位与缩放               |

### 18.4.7 间距、尺寸与视觉节奏

Tailwind 的间距刻度是设计系统的骨架。默认刻度中，`1` 对应 `0.25rem`（4px），`4` 对应 `1rem`（16px），`8` 对应 `2rem`（32px）。

| 方向        | 类名示例                   | 说明                   |
| ----------- | -------------------------- | ---------------------- |
| 水平内边距  | `px-4`、`px-6`             | `padding-left/right`   |
| 垂直内边距  | `py-2`、`py-4`             | `padding-top/bottom`   |
| 四边内边距  | `p-4`、`p-6`               | `padding`              |
| 水平外边距  | `mx-auto`、`mx-4`          | `margin-left/right`    |
| 垂直外边距  | `mt-2`、`mb-4`、`space-y-4` | 单向外边距或子元素间距 |
| 宽度        | `w-full`、`w-64`、`w-1/3`  | 百分比、固定或任意值   |
| 高度        | `h-8`、`h-screen`、`h-full` | 固定、视口或父容器高度 |
| 最小高度    | `min-h-screen`、`min-h-full` | 保证容器撑满           |

视觉节奏的关键是**保持偶数关系**：如果卡片内边距用 `p-6`，卡片之间的间距用 `gap-6` 或 `space-y-6`，整体就会显得整齐。临时插入 `p-[13px]` 或 `m-[7px]` 这类方括号任意值会打破这种节奏，应尽量避免。

`space-x-*` / `space-y-*` 与 `gap-*` 的区别：

- `gap-*` 用于 Flex/Grid 容器，控制所有子元素之间的间隙；
- `space-y-*` 用于普通块级容器，给每个非首子元素加 `margin-top`；
- 优先使用 `gap-*`，因为它不会产生首元素外边距问题，且语义更清晰。

### 18.4.8 过渡、变换与交互状态

界面不是静态的，hover、focus、disabled、loading 都需要样式反馈。Tailwind 通过变体前缀（variants）把这些状态直接表达在类名字符串里。

#### 常用变体前缀

| 变体       | 示例                                | 触发条件                       |
| ---------- | ----------------------------------- | ------------------------------ |
| `hover:`   | `hover:bg-brand-700`                | 鼠标悬停                       |
| `focus:`   | `focus:ring-2 focus:ring-brand-300` | 获得焦点                       |
| `active:`  | `active:bg-brand-800`               | 鼠标按下                       |
| `disabled:`| `disabled:opacity-50`               | 元素 `disabled`                |
| `dark:`    | `dark:bg-slate-800`                 | 父元素含 `dark` 类             |
| `first:` / `last:` | `first:mt-0`、`last:mb-0`   | 第一个 / 最后一个子元素        |
| `odd:` / `even:`   | `odd:bg-gray-50`             | 奇偶行                         |
| `group-hover:` | `group-hover:text-brand-600`    | 父元素标记 `group` 后的悬停    |

#### 过渡与变换

| 能力       | 类名示例                                         | 说明                       |
| ---------- | ------------------------------------------------ | -------------------------- |
| 过渡属性   | `transition-colors`、`transition-all`、`transition-transform` | 指定哪些属性参与过渡       |
| 过渡时长   | `duration-150`、`duration-300`                   | 毫秒为单位的时长           |
| 缓动函数   | `ease-in-out`、`ease-out`                        | 贝塞尔曲线                 |
| 缩放       | `scale-95`、`hover:scale-105`                    | 2D 缩放                    |
| 旋转       | `rotate-45`、`hover:rotate-90`                   | 2D 旋转                    |
| 位移       | `translate-x-2`、`translate-y-full`              | 2D 位移                    |

把交互状态组合起来的典型按钮：

```html
<button class="rounded-lg bg-brand-600 px-4 py-2 font-medium text-white transition-colors duration-200 hover:bg-brand-700 focus:outline-none focus:ring-2 focus:ring-brand-300 active:scale-95 disabled:opacity-50">
  提交
</button>
```

解读：

- `transition-colors duration-200` 让背景色变化在 200ms 内平滑过渡；
- `hover:bg-brand-700` 在悬停时加深背景；
- `focus:outline-none focus:ring-2 focus:ring-brand-300` 用焦点环替代默认轮廓，保证键盘可访问性；
- `active:scale-95` 按下时轻微缩小，给予触觉反馈；
- `disabled:opacity-50` 在禁用时降低不透明度。

## 18.5 暗色模式：class 策略、系统偏好、切换持久化

### 18.5.1 三种暗色策略对比

| 策略    | 配置                   | 特点                               |
| ------- | ---------------------- | ---------------------------------- |
| `media` | `darkMode: 'media'`    | 完全跟随系统偏好，无法手动切换     |
| `class` | `darkMode: 'class'`    | 手动可控，便于持久化和预览         |
| 组合    | `class` + `matchMedia` | 首次按系统，之后按用户选择（推荐） |

我们选择**组合策略**：配置用 `class`，首次加载时读取 `prefers-color-scheme`，用户切换后写入 `localStorage`。

### 18.5.2 useDarkMode Hook

```tsx
// 文件: src/frontend/src/shared/hooks/useDarkMode.ts（教学示例）

import { useEffect, useState } from 'react'

export type Theme = 'light' | 'dark'

export function useDarkMode() {
  const [theme, setTheme] = useState<Theme>('light')

  useEffect(() => {
    const stored = localStorage.getItem('theme') as Theme | null
    if (stored) {
      setTheme(stored)
      return
    }
    if (window.matchMedia('(prefers-color-scheme: dark)').matches) {
      setTheme('dark')
    }
  }, [])

  useEffect(() => {
    const root = document.documentElement
    if (theme === 'dark') {
      root.classList.add('dark')
    } else {
      root.classList.remove('dark')
    }
    localStorage.setItem('theme', theme)
  }, [theme])

  const toggle = () => setTheme((t) => (t === 'light' ? 'dark' : 'light'))

  return { theme, isDark: theme === 'dark', setTheme, toggle }
}
```

关键点：

- 首次挂载时先读 `localStorage`，没有记录再读系统偏好；
- 每次 `theme` 变化时，同步到 `html` 的 `class` 和 `localStorage`；
- `document.documentElement` 就是 `<html>`，这样 `dark:` 变体才能生效。

### 18.5.3 ThemeToggle 组件


## 18.6 性能优化：JIT 模式、CSS 压缩、purge 策略

### 18.6.1 JIT 模式：只生成你写的类

Tailwind v3 默认启用 JIT。开发时，如果你在 JSX 里写：

```html
<div className="w-[123px] bg-brand-600"></div>
```

Tailwind 会即时生成 `w-[123px]` 和 `bg-brand-600` 对应的 CSS，而不是预先生成所有可能值。这带来两个好处：

- 开发构建速度极快；
- 支持任意值语法（如 `w-[123px]`、`bg-[#1da1f2]`），不再受限于预设刻度。

> **回顾**：方括号任意值的写法规则（下划线替换空格、`fr` 表示剩余空间份数）已在 18.3.2 节结合 `grid-cols-[200px_1fr]` 详细讲解。JIT 让这种"临时写 CSS 值"的方式没有性能负担。

### 18.6.2 `content` 即 purge 策略

在 Tailwind v2 时代，purge 是一个独立配置项；v3 把它合并进了 `content`。只要 `content` 路径写全，生产构建时未使用的类自然不会被包含。

```cjs
content: ['./index.html', './src/**/*.{js,ts,jsx,tsx}'],
```

常见漏写路径导致类名失效的场景：

- 模板文件不在 `src` 下（如 `index.html`）；
- 使用动态字符串拼接类名（Tailwind 无法静态分析）；
- 第三方组件库的 JSX 没有被扫描到。

> **提示**：避免用字符串拼接类名。如果需要动态类，优先用 `clsx` / `tailwind-merge` 组合完整类名字符串，或把固定组合抽到组件里。

### 18.6.3 Vite 生产构建自动压缩 CSS

Vite 生产构建会调用 `cssnano` 压缩 CSS。配合 Tailwind 的 JIT，最终 CSS 文件通常只有几 KB 到十几 KB。

```bash
cd src/frontend
pnpm build
ls -lh dist/assets/*.css
```

你可以看到生成的 CSS 只包含实际使用的工具类，没有冗余。

### 18.6.4 性能建议与注意事项

| 建议                    | 原因                                    |
| ----------------------- | --------------------------------------- |
| `content` 范围精确      | 避免扫描无关文件，加快构建              |
| 优先使用预设刻度        | 保持设计一致性，减少任意值滥用          |
| 慎用大面积 `@apply`     | 否则又回到"自定义类名 + 大段 CSS"的老路 |
| 不要把所有颜色都自定义  | 保留 Tailwind 默认灰阶、红绿等语义色    |
| 暗色模式用 `class` 策略 | 便于持久化和精确控制                    |

## 小结

- Tailwind 的 Utility-First 不是反语义化，而是把语义收敛到 `tailwind.config` 这一层，让设计令牌成为团队规范的唯一真相源。
- `tailwind.config.cjs` + `postcss.config.cjs` + `index.css` 是 Vite 项目接入 Tailwind v3 的最小闭环；`main.tsx` 里需要引入 `index.css`。
- `content` 字段决定了 Tailwind 扫描哪些文件生成类名，漏写路径是类名失效的最常见原因。
- Flexbox、Grid、响应式断点用工具类表达，可大幅减少布局样板代码；移动优先的断点策略符合现代设备分布。
- 容器、定位、层级、溢出等工具类让"固定导航 + 居中内容"等常见页面结构无需手写 CSS。
- 设计令牌通过 `theme.extend` 扩展，团队共享同一套颜色、字体、间距、圆角、阴影规范；`Button` 和 `TokenCard` 展示了品牌令牌的实际用法。
- 颜色、文字、边框、背景、间距、尺寸工具类覆盖了绝大多数视觉样式需求；优先使用预设刻度，保持视觉节奏一致。
- 过渡、变换与交互状态变体（`hover:` / `focus:` / `active:` / `disabled:` / `dark:`）让组件在视觉上即时响应用户操作。
- 暗色模式使用 `darkMode: 'class'` 配合 `useDarkMode` Hook，可兼顾系统偏好、用户手动切换和 `localStorage` 持久化。
- 生产构建依赖 JIT 与 `content` 扫描实现按需生成，配合 Vite 的 cssnano 完成压缩；`grid-cols-[200px_1fr]` 等方括号任意值能表达预设类无法覆盖的场景，但反复出现的值应收敛到设计令牌。
