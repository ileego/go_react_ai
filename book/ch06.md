# 第6章 Gin 框架与 RESTful API 设计

前三章搭好了项目的骨架：环境、架构、方法学。从这一章开始，我们进入第二部分——**Go 后端工程化实战**。后端是整个系统的根基，AI 调度和报告生成最终都要落到这里。

Gin 是 Go 社区最流行的 Web 框架，性能高、API 简洁、中间件机制成熟。这一章的核心目标是：**让 API 从"能跑"变成"好维护"**。我们会覆盖路由组织、请求校验、统一响应、中间件开发，以及如何把 Swagger 文档真正嵌入到服务里。

## 6.1 Gin 核心：路由、中间件、上下文的生命周期

### 为什么选 Gin

Go 标准库的 `net/http` 能写 Web 服务，但生产环境需要路由参数提取、请求校验、中间件链、 panic 恢复这些能力。Gin 在这些方面做得最平衡：

- **性能**：基于 `httprouter` 的 radix tree，路由匹配速度在纳秒级
- **中间件**：洋葱模型，通过 `c.Next()` 控制执行顺序
- **上下文**：`gin.Context` 封装了 `http.Request` 和 `http.ResponseWriter`，并提供了参数绑定、JSON 序列化等快捷方法

### 生命周期：一个请求是怎么被处理的

```
请求进入
  → Recovery（捕获 panic）
    → RequestID（注入追踪 ID）
      → Logger（记录请求开始）
        → CORS（设置响应头）
          → 路由匹配 → Handler（业务逻辑）
        ← Logger（记录请求结束、耗时）
      ← RequestID（响应头已设置）
    ← Recovery（无 panic，跳过）
  ← 返回响应
```

中间件的执行顺序是**嵌套**的，不是线性的。先执行的 `Before` 逻辑会后执行 `After` 逻辑，这是洋葱模型的核心。

### 创建 Engine 的两种方式

```go
// 默认：自带 Logger 和 Recovery
r := gin.Default()

// 纯净：不带任何中间件，完全自己控制
r := gin.New()
```

我们选 `gin.New()`，因为日志和恢复要自己实现（后面会讲为什么）。

## 6.2 RESTful API 设计最佳实践

### 资源命名

RESTful 的核心是**资源**，不是动作。URL 应该是名词，动作由 HTTP 方法表达：

| 操作 | HTTP 方法 | URL | 说明 |
|------|-----------|-----|------|
| 创建报告 | POST | `/api/reports` | 集合上创建 |
| 获取列表 | GET | `/api/reports` | 查询集合 |
| 获取详情 | GET | `/api/reports/:id` | 操作单个资源 |
| 取消报告 | POST | `/api/reports/:id/cancel` | 业务动作，用 POST |

几个原则：

1. **用复数名词**：`/reports` 比 `/report` 好，因为集合操作更常见
2. **不用动词**：`/getReport` ❌，`GET /reports/:id` ✅
3. **嵌套不超过两层**：`/api/reports/:id/tasks` ✅，`/api/users/:uid/reports/:rid/tasks/:tid/attachments` ❌（太深了）
4. **业务动作用 POST**：取消、支付、提交这些不是 CRUD 的操作，用 `POST /resource/:id/action`

### 状态码要诚实

不要所有错误都返回 200 然后在 body 里写 `code: 500`。HTTP 状态码就是干这个的：

| 场景 | 状态码 |
|------|--------|
| 成功 | 200 / 201 |
| 参数校验失败 | 400 |
| 未认证 | 401 |
| 无权限 | 403 |
| 资源不存在 | 404 |
| 资源冲突（重复） | 409 |
| 服务器内部错误 | 500 |

### 版本控制

我们的项目目前不大，API 版本直接放在 URL 路径里：

```
/api/v1/reports
/api/v2/reports
```

等到需要维护多个版本时再拆分。前期在 `openapi.yaml` 里标注版本号即可，路由暂时不加 `v1`。

## 6.3 请求参数绑定与自定义校验

### 三种参数来源

Gin 的 `ShouldBind` 系列方法支持从多种来源提取参数：

```go
// JSON body
c.ShouldBindJSON(&req)

// Query 参数
c.ShouldBindQuery(&query)

// URI 路径参数
c.ShouldBindUri(&uri)

// Form 数据
c.ShouldBind(&form)
```

### 结构体标签校验

我们在 `CreateRequest` 里用 `binding` 标签做基础校验：

```go
// 文件: src/backend/internal/handler/report_handler.go
type CreateRequest struct {
    Title string `json:"title" binding:"required,min=1,max=200"`
    Topic string `json:"topic" binding:"required,min=1,max=500"`
}
```

Gin 底层使用 `go-playground/validator`，支持的标签很丰富：

- `required`：必填
- `min`, `max`：字符串长度或数值范围
- `email`：邮箱格式
- `gt`, `lt`：大于 / 小于
- `oneof`：枚举值
- `url`：URL 格式

### 自定义校验

如果内置标签不够用，可以注册自定义校验函数。比如校验研究主题不能包含敏感词：

```go
import "github.com/go-playground/validator/v10"

func init() {
    if v, ok := binding.Validator.Engine().(*validator.Validate); ok {
        v.RegisterValidation("not_banned", func(fl validator.FieldLevel) bool {
            banned := []string{"赌博", "色情"}
            for _, w := range banned {
                if strings.Contains(fl.Field().String(), w) {
                    return false
                }
            }
            return true
        })
    }
}
```

使用方式：

```go
type CreateRequest struct {
    Title string `json:"title" binding:"required,not_banned"`
}
```

### 分页参数的默认值

列表接口的分页参数用 `DefaultQuery` 处理：

```go
page, _ := strconv.Atoi(c.DefaultQuery("page", "1"))
pageSize, _ := strconv.Atoi(c.DefaultQuery("page_size", "20"))
```

注意：Gin 的 `DefaultQuery` 返回字符串，需要手动转换。如果转换失败，直接用默认值兜底。

## 6.4 统一响应封装：成功、错误、分页的标准结构

### 为什么必须统一

前后端协作最痛苦的事情之一，是同一个项目里响应格式五花八门：

```json
// A 接口
{"code": 0, "data": {...}}

// B 接口
{"status": "ok", "result": {...}}

// C 接口出错
{"error": "something wrong"}

// D 接口出错
{"message": "something wrong", "status": 500}
```

前端为了适配这些格式，不得不写一堆条件判断。统一响应结构后，前端可以用一个拦截器处理所有请求。

### 我们的响应结构

```go
// 文件: src/backend/pkg/response/response.go

// Body 统一响应体
type Body[T any] struct {
    Code    int    `json:"code"`
    Message string `json:"message,omitempty"`
    Data    T      `json:"data,omitempty"`
}

// ListBody 列表响应体
type ListBody[T any] struct {
    Code    int    `json:"code"`
    Message string `json:"message,omitempty"`
    Data    T      `json:"data,omitempty"`
    Total   int    `json:"total,omitempty"`
    Page    int    `json:"page,omitempty"`
    Size    int    `json:"size,omitempty"`
}
```

成功时 `Code` 为 `0`，错误时为 HTTP 状态码（方便前端做判断）。`Message` 只在出错时填充。

### 响应辅助函数

```go
// 成功响应（无数据）
response.OK(c)
// → {"code": 0}

// 成功响应（有数据）
response.Data(c, report)
// → {"code": 0, "data": {...}}

// 创建成功
response.Created(c, report)
// → {"code": 0, "data": {...}}  HTTP 201

// 列表响应
response.List(c, reports, total, page, pageSize)
// → {"code": 0, "data": [...], "total": 100, "page": 1, "size": 20}

// 错误响应
response.BadRequest(c, "标题不能为空")
// → {"code": 400, "message": "标题不能为空"}  HTTP 400
```

### 从业务错误自动映射 HTTP 状态码

我们在 `pkg/errors` 里定义了错误类型（`KindValidation`、`KindNotFound` 等），现在让 `response` 包自动映射：

```go
// 文件: src/backend/pkg/response/response.go

func FromError(c *gin.Context, err error) {
    if e, ok := err.(*errors.Error); ok {
        switch e.Kind {
        case errors.KindValidation:
            Error(c, http.StatusBadRequest, e.Message)
        case errors.KindNotFound:
            Error(c, http.StatusNotFound, e.Message)
        case errors.KindDuplicate:
            Error(c, http.StatusConflict, e.Message)
        case errors.KindUnauthorized:
            Error(c, http.StatusUnauthorized, e.Message)
        case errors.KindForbidden:
            Error(c, http.StatusForbidden, e.Message)
        default:
            Error(c, http.StatusInternalServerError, e.Message)
        }
        return
    }
    InternalServerError(c, err.Error())
}
```

这样 Handler 层处理错误就一行代码：

```go
report, err := h.svc.GetByID(c.Request.Context(), id)
if err != nil {
    response.FromError(c, err)  // 自动映射 404 / 500
    return
}
response.Data(c, report)
```

## 6.5 中间件开发：日志、恢复、跨域、请求追踪 ID

### 中间件的本质

中间件就是一个 `gin.HandlerFunc`，它在请求处理前后插入逻辑。Gin 通过 `c.Next()` 把控制权交给下一个中间件或 Handler，`c.Next()` 返回后继续执行当前中间件的后续逻辑。

### Recovery：最后一道防线

生产环境绝对不能因为某个 Handler 的 panic 就让整个服务崩溃。Gin 内置了 `gin.Recovery()`，但我们自己实现一个，增加结构化日志和请求 ID：

```go
// 文件: src/backend/internal/middleware/recovery.go

func Recovery() gin.HandlerFunc {
    return func(c *gin.Context) {
        defer func() {
            if err := recover(); err != nil {
                slog.Error("panic recovered",
                    slog.Any("error", err),
                    slog.String("path", c.Request.URL.Path),
                    slog.String("method", c.Request.Method),
                    slog.String("request_id", GetRequestID(c)),
                )
                response.InternalServerError(c, "服务器内部错误")
                c.Abort()
            }
        }()
        c.Next()
    }
}
```

注意 `c.Abort()`：panic 发生后必须阻止后续 Handler 执行，否则可能产生不可预期的副作用。

### Logger：结构化请求日志

不要用 `fmt.Println` 或 `log.Printf` 打印请求日志。Go 1.21 引入的 `log/slog` 支持结构化输出，方便后续接入日志聚合系统（如 Loki、ELK）：

```go
// 文件: src/backend/internal/middleware/logger.go

func Logger() gin.HandlerFunc {
    return func(c *gin.Context) {
        start := time.Now()
        path := c.Request.URL.Path
        raw := c.Request.URL.RawQuery

        c.Next()  // 执行后续 Handler

        latency := time.Since(start)
        status := c.Writer.Status()

        if raw != "" {
            path = path + "?" + raw
        }

        slog.Info("http request",
            slog.String("method", c.Request.Method),
            slog.String("path", path),
            slog.Int("status", status),
            slog.Duration("latency", latency),
            slog.String("ip", c.ClientIP()),
            slog.String("request_id", GetRequestID(c)),
            slog.Int("errors", len(c.Errors)),
        )
    }
}
```

在 `main.go` 中初始化 slog：

```go
slog.SetDefault(slog.New(slog.NewJSONHandler(os.Stdout, nil)))
```

这样日志输出是 JSON 格式，可以直接被日志系统解析：

```json
{"time":"2026-06-09T10:30:00Z","level":"INFO","msg":"http request","method":"GET","path":"/api/reports","status":200,"latency":"5.2ms","ip":"127.0.0.1","request_id":"a3f9e2d1","errors":0}
```

### RequestID：分布式追踪的起点

没有请求 ID 时，排查线上问题就像大海捞针。一个请求可能经过网关、后端、数据库、缓存，如果每个环节都打印相同的 ID，就能把整个调用链串起来。

```go
// 文件: src/backend/internal/middleware/request_id.go

const HeaderXRequestID = "X-Request-ID"

func RequestID() gin.HandlerFunc {
    return func(c *gin.Context) {
        rid := c.GetHeader(HeaderXRequestID)
        if rid == "" {
            rid = generateID()  // 8 字节随机 hex
        }
        c.Set(HeaderXRequestID, rid)
        c.Writer.Header().Set(HeaderXRequestID, rid)
        c.Next()
    }
}
```

如果前端或网关已经传了 `X-Request-ID`，就复用它，保证整条链路的 ID 一致。

### CORS：跨域中间件

之前的 CORS 直接写在 `main.go` 里，现在提取成可复用的中间件，支持多来源配置：

```go
// 文件: src/backend/internal/middleware/cors.go

func CORS(allowOrigins string) gin.HandlerFunc {
    return func(c *gin.Context) {
        origin := c.Request.Header.Get("Origin")
        if allowOrigins == "*" || containsOrigin(allowOrigins, origin) {
            c.Writer.Header().Set("Access-Control-Allow-Origin", origin)
        }
        c.Writer.Header().Set("Access-Control-Allow-Methods", "GET, POST, PUT, DELETE, PATCH, OPTIONS")
        c.Writer.Header().Set("Access-Control-Allow-Headers", "Content-Type, Authorization, X-Request-ID")
        c.Writer.Header().Set("Access-Control-Expose-Headers", "X-Request-ID")
        c.Writer.Header().Set("Access-Control-Max-Age", "86400")

        if c.Request.Method == http.MethodOptions {
            c.AbortWithStatus(http.StatusNoContent)
            return
        }
        c.Next()
    }
}
```

注意 `Access-Control-Expose-Headers`：前端 JavaScript 默认只能读取 `Content-Type` 等简单响应头，如果前端需要读取我们返回的 `X-Request-ID`，必须显式暴露。

### 中间件顺序

顺序不是随便排的。正确的顺序是：

```go
r.Use(middleware.Recovery())   // 1. 最外层，捕获所有 panic
r.Use(middleware.RequestID())  // 2. 尽早注入，后续中间件和 Handler 都能用
r.Use(middleware.Logger())     // 3. 记录请求开始
r.Use(middleware.CORS(...))    // 4. 设置响应头
```

Logger 放在 RequestID 之后，才能保证日志里有 `request_id`。Recovery 必须放在最外层，才能捕获 Logger 或 CORS 中可能出现的 panic。

## 6.6 路由分组与模块化注册

### 按资源分组

路由一多就会乱。Gin 的 `Group` 可以把同一资源的接口归到一起：

```go
// 文件: src/backend/internal/handler/handler.go

func (h *Handlers) RegisterRoutes(r *gin.Engine) {
    api := r.Group("/api")
    {
        api.GET("/health", h.Health.Check)

        reports := api.Group("/reports")
        {
            reports.POST("", h.Report.Create)
            reports.GET("", h.Report.List)
            reports.GET("/:id", h.Report.Get)
            reports.POST("/:id/cancel", h.Report.Cancel)
            reports.POST("/:report_id/dispatch", h.Agent.Dispatch)
        }
    }
}
```

分组的好处：
- 结构清晰，一眼看出有哪些资源
- 后续可以给分组统一加中间件（如认证）
- 方便生成 OpenAPI 文档时按资源组织

### 依赖注入

Handler 层不直接创建 Service，而是通过构造函数注入：

```go
// main.go 中的组装
reportRepo := memory.NewReportRepository()
reportSvc := service.NewReportService(reportRepo)
agentSvc := service.NewAgentService(reportRepo, nil)

handlers := handler.NewHandlers(reportSvc, agentSvc)
handlers.RegisterRoutes(r)
```

这种"在外部组装、在内部使用"的模式，是 Clean Architecture 的核心实践。第7章把 `memory.NewReportRepository()` 换成 PostgreSQL 实现时，Handler 和 Service 的代码一行都不需要改。

### Swagger 文档：生成、嵌入与维护

第3章讲过，本书采用 **Contract-First** 方式：先手写 `openapi.yaml`，前后端再各自实现。这里补充一下这份契约是怎么产生、怎么维护、怎么和代码保持同步的。

#### OpenAPI 文档的三种来源

1. **手写**：开发者在 `docs/openapi.yaml` 里直接写契约。优点是精确可控，缺点是容易忘记更新。
2. **代码注释生成**：用 `swaggo/swag` 等工具，在 Handler 函数上方写注释，自动生成 OpenAPI。优点是和代码同步，缺点是注释一大就会污染代码。
3. **运行时反射生成**：部分框架能根据结构体标签自动生成。优点是完全自动化，缺点是控制力弱，复杂场景不好表达。

我们的项目选 **手写 + 嵌入 Swagger UI** 的组合。原因很务实：

- 接口数量可控，手写的维护成本不高
- 契约比代码先定下来，方便前后端并行开发
- `openapi.yaml` 本身就是一个审查点，PR 时必须检查它是否与代码一致

#### 维护契约的工作流

1. **需求确定后**，先讨论接口：URL、方法、请求参数、响应结构、错误码
2. **在 `docs/openapi.yaml` 中新增/修改**接口定义
3. **启动后端服务**，访问 `/swagger` 用 Swagger UI 预览渲染效果
4. **前端执行 `openapi-typescript`** 重新生成 TypeScript 类型
5. **前后端各自实现**，联调时以 OpenAPI 文件为准

```bash
# 前端生成类型（本书项目结构）
$ cd src/frontend
$ npx openapi-typescript ../backend/docs/openapi.yaml -o src/shared/api/types.gen.ts
```

#### 常用辅助工具

| 工具 | 用途 | 适用场景 |
|------|------|----------|
| [Swagger Editor](https://editor.swagger.io/) | 在线编辑、实时预览 OpenAPI | 本地没有 UI 时快速验证 |
| [Prism](https://stoplight.io/open-source/prism) | 根据 OpenAPI 启动 Mock 服务 | 前端先开发，后端接口还没就绪 |
| [openapi-typescript](https://github.com/drwpow/openapi-typescript) | 生成 TypeScript 类型 | 前端类型安全 |
| [swaggo/swag](https://github.com/swaggo/swag) | 从 Go 注释生成 OpenAPI | 偏好代码驱动文档的团队 |

#### Swagger UI 嵌入

第3章承诺过"第6章会介绍如何在 Gin 中嵌入 Swagger UI"。现在兑现：

```go
// 文件: src/backend/docs/swagger.go

//go:embed openapi.yaml
var openapiYAML []byte

//go:embed swagger-ui.html
var swaggerUIHTML []byte

func RegisterRoutes(r *gin.Engine) {
    r.GET("/swagger", func(c *gin.Context) {
        c.Redirect(http.StatusMovedPermanently, "/swagger/index.html")
    })
    r.GET("/swagger/index.html", func(c *gin.Context) {
        c.Data(http.StatusOK, "text/html; charset=utf-8", swaggerUIHTML)
    })
    r.GET("/swagger/openapi.yaml", func(c *gin.Context) {
        c.Data(http.StatusOK, "application/yaml", openapiYAML)
    })
}
```

`go:embed` 把静态文件编译进二进制，部署时不需要额外拷贝文件。启动服务后访问 `http://localhost:8080/swagger` 即可看到交互式文档。

### 启动后的完整 main.go

```go
// 文件: src/backend/cmd/server/main.go

func main() {
    cfg := config.Load()

    if cfg.Server.Mode == "release" {
        gin.SetMode(gin.ReleaseMode)
    }

    slog.SetDefault(slog.New(slog.NewJSONHandler(os.Stdout, nil)))

    // 依赖注入
    reportRepo := memory.NewReportRepository()
    reportSvc := service.NewReportService(reportRepo)
    agentSvc := service.NewAgentService(reportRepo, nil)
    handlers := handler.NewHandlers(reportSvc, agentSvc)

    r := gin.New()
    r.Use(middleware.Recovery())
    r.Use(middleware.RequestID())
    r.Use(middleware.Logger())
    r.Use(middleware.CORS(cfg.Server.AllowOrigins))

    handlers.RegisterRoutes(r)
    docs.RegisterRoutes(r)

    addr := ":" + cfg.Server.Port
    log.Printf("Server starting on %s (mode=%s)", addr, cfg.Server.Mode)
    if err := r.Run(addr); err != nil {
        log.Fatalf("Server failed: %v", err)
    }
}
```

## 小结

这一章把 API 的"面子"和"里子"都做了：

- **面子**：统一的响应结构、正确的 HTTP 状态码、RESTful 资源命名
- **里子**：洋葱模型中间件、结构化日志、请求追踪、panic 恢复、依赖注入

```bash
$ curl -i http://localhost:8080/api/health

X-Request-ID: 3f2a9c1d
Content-Type: application/json; charset=utf-8

{"code":0,"data":{"status":"ok","time":175...


$ curl -i -X POST http://localhost:8080/api/reports \
  -H "Content-Type: application/json" \
  -d '{"title":"AI 发展趋势","topic":"人工智能"}'

```

响应里的 `X-Request-ID` 就是我们在 RequestID 中间件里生成的。如果后续出了问题，拿着这个 ID 去日志里一搜，就能找到这条请求的全链路记录。
