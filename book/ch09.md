# 第9章 用户系统、认证与授权

第8章我们把可观测性补全了，但一个面向多用户的应用如果没有认证授权，就像一座没有门禁的大楼。本章我们实现完整的用户系统：JWT 双令牌认证、bcrypt 密码、RBAC 权限、Redis 速率限制与 Token 黑名单，以及 GitHub OAuth2 第三方登录。

这些能力不仅是当前报告平台的刚需，也是后续多智能体协作、资源隔离和审计追踪的基础。

## 9.1 JWT 完整方案：Access Token + Refresh Token 双令牌

JWT（JSON Web Token）是前后端分离系统里最常见的无状态认证凭证。它的优点是服务端不需要保存会话，缺点是令牌一旦签发就很难提前吊销。生产环境通常采用**双令牌策略**：

- **Access Token**：短期有效（默认 15 分钟），携带用户身份，用于访问受保护接口。
- **Refresh Token**：长期有效（默认 7 天），仅用于换取新的 Access Token，不直接访问业务接口。

这样做的好处是：即使 Access Token 泄露，攻击窗口也只有十几分钟；Refresh Token 的泄露风险虽然更大，但它只在刷新接口使用，且可以通过黑名单吊销。

### Claims 设计

我们在 `internal/auth/jwt.go` 中定义自定义 Claims：

```go
// 文件: src/backend/internal/auth/jwt.go

type TokenType string

const (
    TokenTypeAccess  TokenType = "access"
    TokenTypeRefresh TokenType = "refresh"
)

type Claims struct {
    UserID int64     `json:"uid"`
    Role   string    `json:"role"`
    Type   TokenType `json:"type"`
    jwt.RegisteredClaims
}
```

`UserID` 和 `Role` 是业务最关心的两个字段；`Type` 区分 access 与 refresh，避免用 refresh token 去调业务接口；`jwt.RegisteredClaims` 提供 `exp`、`iat`、`nbf`、`iss`、`sub`、`jti` 等标准声明。`jti`（JWT ID）是吊销令牌的关键：我们把要吊销的 `jti` 写入 Redis 黑名单。

### 生成与验证

```go
// 文件: src/backend/internal/auth/jwt.go

func GenerateTokenPair(cfg Config, userID int64, role string) (*TokenPair, error) {
    accessJTI := uuid.NewString()
    refreshJTI := uuid.NewString()

    accessToken, err := generateToken(cfg, userID, role, TokenTypeAccess, accessJTI, cfg.AccessTokenTTL)
    if err != nil {
        return nil, fmt.Errorf("generate access token: %w", err)
    }

    refreshToken, err := generateToken(cfg, userID, role, TokenTypeRefresh, refreshJTI, cfg.RefreshTokenTTL)
    // ...
}

func ParseAndValidate(tokenString string, expectedType TokenType, secret string) (*Claims, error) {
    token, err := jwt.ParseWithClaims(tokenString, &Claims{}, func(token *jwt.Token) (any, error) {
        if _, ok := token.Method.(*jwt.SigningMethodHMAC); !ok {
            return nil, fmt.Errorf("unexpected signing method: %v", token.Header["alg"])
        }
        return []byte(secret), nil
    })
    // ...
}
```

生成时使用 `jwt.SigningMethodHS256`（HMAC + SHA-256），验证时检查签名算法，防止 `alg: none` 攻击。

### Refresh Token Rotation

刷新令牌不是简单地延长旧 Access Token 的有效期，而是**签发一对全新的令牌**，并把旧的 refresh token 加入黑名单：

```go
// 文件: src/backend/internal/service/auth_service.go

func (s *authService) Refresh(ctx context.Context, refreshToken string) (string, string, error) {
    claims, err := auth.ParseAndValidate(refreshToken, auth.TokenTypeRefresh, s.jwtCfg.Secret)
    // 检查黑名单 ...

    pair, err := auth.GenerateTokenPair(s.jwtCfg, claims.UserID, claims.Role)
    // 将旧的 refresh token 加入黑名单
    _ = s.blacklist.Add(ctx, claims.ID, s.jwtCfg.RefreshTokenTTL)
    return pair.AccessToken, pair.RefreshToken, nil
}
```

这实现了 Refresh Token Rotation：每次刷新后旧 refresh token 失效，即使被盗也无法反复使用。

## 9.2 密码安全：bcrypt 工作因子选择与强度校验

不要自己写密码哈希。Go 生态里 `golang.org/x/crypto/bcrypt` 是标准选择。它的 `cost` 参数决定计算强度：cost 越大，破解越慢，但登录也越慢。默认值 10 对大多数应用偏低，我们选择 **cost = 12**，在普通笔记本上哈希一次约 200ms，既能抗暴力破解，又不影响用户体验。

```go
// 文件: src/backend/internal/service/auth_service.go

const bcryptCost = 12

func (s *authService) Register(ctx context.Context, email, password, nickname string) (*domain.User, error) {
    // ...
    hash, err := bcrypt.GenerateFromPassword([]byte(password), bcryptCost)
    // ...
}

func (s *authService) Login(ctx context.Context, email, password string) (string, string, error) {
    user, hash, err := s.userRepo.GetByEmailWithPassword(ctx, email)
    // ...
    if err := bcrypt.CompareHashAndPassword([]byte(hash), []byte(password)); err != nil {
        // 登录失败，记录失败次数
        return "", "", apperrors.NewUnauthorized("邮箱或密码错误")
    }
}
```

密码强度校验也是 Service 层的责任。我们的策略是：最少 8 位，且同时包含大写字母、小写字母和数字。

```go
func validatePassword(password string) error {
    if len(password) < minPasswordLength {
        return fmt.Errorf("密码长度至少 %d 位", minPasswordLength)
    }
    var hasUpper, hasLower, hasDigit bool
    for _, r := range password {
        switch {
        case unicode.IsUpper(r):
            hasUpper = true
        case unicode.IsLower(r):
            hasLower = true
        case unicode.IsDigit(r):
            hasDigit = true
        }
    }
    if !hasUpper || !hasLower || !hasDigit {
        return errors.New("密码必须同时包含大写字母、小写字母和数字")
    }
    return nil
}
```

> **注意**：bcrypt 有 72 字节的输入限制，超长密码会被截断。如果业务需要支持超长密码，可以在 bcrypt 之前先做一次 SHA-256 哈希。

## 9.3 认证中间件：Token 解析、上下文注入、黑名单

认证中间件负责三件事：从 `Authorization: Bearer <token>` 中解析 token、校验 token 是否有效且未被吊销、把用户身份注入 gin 上下文。

```go
// 文件: src/backend/internal/middleware/auth.go

func JWTAuth(secret string, blacklist security.TokenBlacklist) gin.HandlerFunc {
    return func(c *gin.Context) {
        tokenString := extractBearerToken(c.GetHeader("Authorization"))
        if tokenString == "" {
            response.FromError(c, apperrors.NewUnauthorized("缺少认证令牌").WithCode("MISSING_TOKEN"))
            c.Abort()
            return
        }

        claims, err := auth.ParseAndValidate(tokenString, auth.TokenTypeAccess, secret)
        if err != nil {
            // 区分过期与无效 ...
        }

        // 黑名单检查
        if blacklist != nil {
            blacklisted, err := blacklist.IsBlacklisted(c.Request.Context(), claims.ID)
            if blacklisted {
                response.FromError(c, apperrors.NewUnauthorized("令牌已被吊销").WithCode("TOKEN_REVOKED"))
                c.Abort()
                return
            }
        }

        c.Set(userIDContextKey, claims.UserID)
        c.Set(userRoleContextKey, claims.Role)
        c.Next()
    }
}
```

上下文注入后，Handler 和 Service 可以通过工具函数取当前用户：

```go
func GetUserID(c *gin.Context) int64   { /* ... */ }
func GetUserRole(c *gin.Context) string { /* ... */ }
```

### Token 黑名单

登出或刷新后，旧令牌需要失效。我们用 Redis 存储 `jti`：

```go
// 文件: src/backend/internal/security/blacklist.go

func (b *RedisTokenBlacklist) Add(ctx context.Context, jti string, ttl time.Duration) error {
    key := fmt.Sprintf("%s:%s", b.keyPrefix, jti)
    return b.client.Set(ctx, key, "1", ttl).Err()
}

func (b *RedisTokenBlacklist) IsBlacklisted(ctx context.Context, jti string) (bool, error) {
    key := fmt.Sprintf("%s:%s", b.keyPrefix, jti)
    n, err := b.client.Exists(ctx, key).Result()
    return n > 0, err
}
```

使用 `SETEX` 让 Redis 自动清理过期条目，不需要额外任务。

## 9.4 RBAC 权限模型：用户、角色、权限、资源四级设计

RBAC（Role-Based Access Control）把权限绑定到角色，再把角色分配给用户。我们的平台定义三种角色：

| 角色 | 说明 | 典型权限 |
|------|------|----------|
| `system` | 系统管理员 | 全部权限（`*`） |
| `admin` | 普通管理员 | 派发报告、读取任意报告、管理用户角色 |
| `user` | 普通用户 | 创建报告、读取自己的报告、取消自己的报告 |

```go
// 文件: src/backend/internal/domain/user.go

type UserRole string

const (
    UserRoleSystem UserRole = "system"
    UserRoleAdmin  UserRole = "admin"
    UserRoleUser   UserRole = "user"
)
```

权限用字符串表示，格式为 `资源:操作[:范围]`，例如 `reports:read:own`、`reports:dispatch`。中间件里通过角色-权限映射表做判断：

```go
// 文件: src/backend/internal/middleware/auth.go

var rolePermissions = map[string][]string{
    "system": {"*"},
    "admin":  {"reports:dispatch", "reports:read:any", "users:read:any", "users:role:update"},
    "user":   {"reports:create", "reports:read:own", "reports:cancel:own"},
}

func RequirePermission(perm string) gin.HandlerFunc {
    return func(c *gin.Context) {
        role := GetUserRole(c)
        if hasPermission(role, perm) {
            c.Next()
            return
        }
        response.FromError(c, apperrors.NewForbidden("权限不足"))
        c.Abort()
    }
}
```

## 9.5 API 鉴权：基于资源和操作的细粒度控制

路由注册时，我们把认证和权限中间件按组应用，而不是在每个 Handler 里重复判断：

```go
// 文件: src/backend/internal/handler/handler.go

reports := api.Group("/reports")
reports.Use(middleware.JWTAuth(h.jwtSecret, h.blacklist))
reports.Use(middleware.RateLimit(h.rl, 100, time.Minute))
{
    reports.POST("", h.Report.Create)
    reports.GET("", h.Report.List)
    reports.GET("/:id", h.Report.Get)
    reports.POST("/:id/cancel", h.Report.Cancel)
    reports.POST("/:report_id/dispatch", middleware.RequireRole("admin", "system"), h.Agent.Dispatch)
}
```

报告列表只能看自己的，报告详情则需要做所有权校验：

```go
// 文件: src/backend/internal/handler/report_handler.go

func (h *ReportHandler) Get(c *gin.Context) {
    report, err := h.svc.GetByID(c.Request.Context(), id)
    // ...

    userID := middleware.GetUserID(c)
    role := middleware.GetUserRole(c)
    if report.CreatedBy != userID && !canReadAnyReport(role) {
        response.FromError(c, apperrors.NewForbidden("无权访问该报告"))
        return
    }

    response.Data(c, report)
}
```

这种"路由级 + Handler 级"两层控制是常见的做法：路由级拦截无权限请求，Handler 级处理资源所有权。

## 9.6 安全加固：速率限制、登录失败锁定、Token 自动续期

### 全局 IP 限流

我们用 Redis 固定窗口计数器限制单个 IP 的请求频率：

```go
// 文件: src/backend/internal/security/ratelimit.go

func (r *RedisRateLimiter) AllowIP(ctx context.Context, key string) (bool, error) {
    redisKey := fmt.Sprintf("%s:ip:%s", r.keyPrefix, key)
    return r.allow(ctx, redisKey, r.ipCfg.Limit, r.ipCfg.Window)
}

func (r *RedisRateLimiter) allow(ctx context.Context, key string, limit int, window time.Duration) (bool, error) {
    pipe := r.client.Pipeline()
    incr := pipe.Incr(ctx, key)
    pipe.Expire(ctx, key, window)
    _, err := pipe.Exec(ctx)
    // ...
    return int(incr.Val()) <= limit, nil
}
```

使用 Pipeline 把 `INCR` 和 `EXPIRE` 打包发送，避免竞态窗口。

### 登录失败锁定

对登录接口，我们单独维护失败计数和锁定状态：

```go
func (r *RedisRateLimiter) RecordLoginFailure(ctx context.Context, key string) (int, time.Duration, error) {
    counterKey := fmt.Sprintf("%s:login:fail:%s", r.keyPrefix, key)
    lockKey := fmt.Sprintf("%s:login:lock:%s", r.keyPrefix, key)

    pipe := r.client.Pipeline()
    incr := pipe.Incr(ctx, counterKey)
    pipe.Expire(ctx, counterKey, r.loginCfg.Window)
    getLock := pipe.TTL(ctx, lockKey)
    _, err := pipe.Exec(ctx)

    count := int(incr.Val())
    if count >= r.loginCfg.MaxAttempts {
        _ = r.client.Set(ctx, lockKey, "1", r.loginCfg.Lockout).Err()
    }
    // ...
}
```

配置为 15 分钟内失败 5 次即锁定 15 分钟。登录成功后调用 `ResetLoginFailures` 清除计数。登录路由会额外挂一个 `middleware.LoginRateLimit` 中间件，在 Service 层处理之前就先检查锁定状态。

### Token 自动续期策略

前端通常这样处理：

1. 把 Access Token 和 Refresh Token 一起存储（推荐内存，次选 httpOnly cookie）。
2. 每次请求携带 Access Token。
3. 收到 401 且 err_code 为 `TOKEN_EXPIRED` 时，用 Refresh Token 调用 `/api/auth/refresh`。
4. 刷新成功则重放原请求；刷新失败则跳转登录页。

后端在 Refresh 时做 Rotation，保证旧 refresh token 一次性有效。

## 9.7 OAuth2.0：GitHub 第三方登录接入

OAuth2.0 让用户无需注册密码就能登录。我们以 GitHub 为例，接入流程如下：

1. 用户点击"GitHub 登录"，后端生成随机 `state`，存入 Redis（5 分钟 TTL），然后重定向到 GitHub 授权页。
2. 用户授权后，GitHub 重定向回 `/api/auth/github/callback?code=xxx&state=yyy`。
3. 后端校验 `state`，用 `code` 换取 GitHub access token。
4. 调用 GitHub `/user` API 获取用户基本信息（id、login、email、avatar_url）。
5. 根据 `github_id` 查找本地用户：存在则登录，不存在则创建或绑定。

```go
// 文件: src/backend/internal/service/auth_service.go

func (s *authService) GithubLogin(ctx context.Context, code string) (string, string, error) {
    token, err := s.oauthCfg.Exchange(ctx, code)
    // ...

    githubUser, err := s.fetchGithubUser(ctx, token.AccessToken)
    // ...

    user, err := s.userRepo.GetByGithubID(ctx, githubID)
    if user == nil {
        existing, _ := s.userRepo.GetByEmail(ctx, githubUser.Email)
        if existing != nil {
            // 绑定 GitHub 到已有账号
            _ = s.userRepo.UpdateGithubInfo(ctx, existing.ID, githubID, githubUser.AvatarURL)
            user = existing
        } else {
            // 创建新用户
            newUser := &domain.User{...}
            _ = s.userRepo.CreateWithGithub(ctx, newUser, githubID)
            user = newUser
        }
    }

    return auth.GenerateTokenPair(s.jwtCfg, user.ID, string(user.Role))
}
```

绑定策略的选择很重要：如果用户先用邮箱注册了账号，之后用同一邮箱的 GitHub 登录，我们应该把它**绑定到已有账号**，而不是创建新账号导致数据分裂。

### 配置项

```bash
# .env.example
JWT_SECRET=change-me-in-production
ACCESS_TOKEN_TTL_MINUTES=15
REFRESH_TOKEN_TTL_DAYS=7
GITHUB_CLIENT_ID=your-github-app-client-id
GITHUB_CLIENT_SECRET=your-github-app-secret
GITHUB_REDIRECT_URL=http://localhost:8080/api/auth/github/callback
```

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `JWT_SECRET` | `change-me-in-production` | 必须替换为强随机字符串 |
| `ACCESS_TOKEN_TTL_MINUTES` | 15 | Access Token 有效期 |
| `REFRESH_TOKEN_TTL_DAYS` | 7 | Refresh Token 有效期 |
| `GITHUB_CLIENT_ID` | - | GitHub OAuth App 客户端 ID |
| `GITHUB_CLIENT_SECRET` | - | GitHub OAuth App 密钥 |
| `GITHUB_REDIRECT_URL` | - | 必须与 GitHub 后台配置一致 |

## 小结

本章我们为平台构建了完整的认证授权体系：

- **JWT 双令牌**：Access Token 短期访问，Refresh Token 长期刷新，配合 Rotation 提升安全性。
- **密码安全**：bcrypt cost 12 + 强度校验，避免自己实现哈希。
- **认证中间件**：Bearer 解析、上下文注入、黑名单校验。
- **RBAC**：system / admin / user 三级角色，权限矩阵控制资源访问。本章采用静态角色映射保证简洁；动态权限（运行时增删角色、策略引擎）将在第 38 章通过 Casbin/OPA 展开。
- **安全加固**：Redis IP 限流、登录失败锁定、Token 黑名单自动过期。
- **GitHub OAuth2**：完整登录/回调流程，支持新用户创建和已有账号绑定。
